# 业务数据库与接口的关联机制说明

## 概述

系统通过多层次的智能分析，将业务数据库的表结构与API接口进行关联，从而：
1. 识别接口需要哪些数据库表的数据
2. 分析接口之间的依赖关系
3. 生成更准确的测试数据

## 关联机制（多步骤流程）

### 第一步：数据库元数据提取

**操作位置**：系统界面的"数据库信息"页面 → 点击"分析数据库"按钮

**执行流程**：
1. 连接业务数据库
2. 分析数据库结构（表、字段、外键、索引等）
3. 提取表注释、字段注释（如果有）
4. 分析表与表之间的关系（外键关系）
5. 将元数据保存到系统数据库的以下表中：
   - `table_metadata`：表元数据（表名、注释、主键、外键等）
   - `column_metadata`：字段元数据（字段名、类型、是否主键/外键、注释等）
   - `table_relationships`：表关系元数据（表之间的关系类型）

**代码位置**：
- `backend/app/routers/db_connections.py` → `analyze_database` 接口
- `backend/app/services/metadata_service.py` → `DatabaseMetadataManager`

### 第二步：构建知识图谱

**执行流程**：
1. 将数据库元数据转换为Neo4j图数据库的节点和关系
2. 在Neo4j中构建知识图谱：
   - 节点：数据库表（Table节点）
   - 节点：字段（Column节点，可选）
   - 关系：表与表之间的关系（HAS_A、DEPENDS_ON等）

**代码位置**：
- `backend/app/services/db_service.py` → `build_knowledge_graph` 方法

### 第三步：接口参数分析与表关联

当分析接口依赖关系时，系统会：

#### 3.1 提取接口参数

从接口的请求参数中提取字段名：
- URL查询参数（params）
- 请求体（request_body）
- 请求头（headers，如Authorization）

#### 3.2 字段名推断表名

系统通过以下方式推断接口需要的表：

**方式1：基于字段名匹配**
- 如果参数包含 `user_id`、`userId`、`user_id` 等 → 推断需要 `users` 表
- 如果参数包含 `device_id`、`deviceId` → 推断需要 `devices` 表
- 如果参数包含 `course_id`、`courseId` → 推断需要 `courses` 表
- 如果参数包含 `*_id` 格式 → 推断对应 `*` 表

**方式2：基于接口URL和名称匹配**
- URL路径包含 `/user` → 关联 `users` 表
- URL路径包含 `/device` → 关联 `devices` 表
- 接口名称包含"用户" → 关联 `users` 表
- 接口名称包含"设备" → 关联 `devices` 表

**代码位置**：
- `backend/app/services/dependency_analyzer.py` → `_analyze_data_dependencies` 方法
- `backend/app/services/dependency_analyzer.py` → `_find_data_source` 方法
- `backend/app/services/smart_test_data_generator.py` → `_infer_tables_from_api` 方法

#### 3.3 通过知识图谱查找关联表

系统从Neo4j知识图谱中查询：
- 如果接口需要表A的数据
- 通过外键关系找到表A依赖的表B
- 从而推断接口可能还需要表B的数据

### 第四步：测试数据生成

当生成测试数据时：

1. **推断相关表**：根据接口信息推断需要的表
2. **获取表元数据**：从系统数据库读取表结构信息
3. **生成测试数据**：
   - 根据字段类型生成对应格式的数据
   - 根据字段约束（主键、外键、非空等）生成合法数据
   - 根据外键关系，从关联表获取真实数据
   - 根据数据特征（如果已分析）生成更真实的数据

**代码位置**：
- `backend/app/services/smart_test_data_generator.py` → `generate_test_data` 方法
- `backend/app/services/smart_test_data_generator.py` → `_get_table_metadata` 方法

## 示例说明

### 示例1：用户注册接口

**接口信息**：
- URL: `POST /api/users/register`
- 请求体: `{"username": "test", "email": "test@example.com", "password": "123456"}`

**关联过程**：
1. URL包含 `/users` → 推断关联 `users` 表
2. 系统查找 `users` 表的元数据（字段类型、约束等）
3. 生成测试数据时，使用 `users` 表的结构生成符合要求的用户名、邮箱等

### 示例2：查看运动记录接口

**接口信息**：
- URL: `GET /api/records?user_id=123&device_id=456`
- 需要认证（Authorization header）

**关联过程**：
1. 参数包含 `user_id` → 推断需要 `users` 表数据
2. 参数包含 `device_id` → 推断需要 `devices` 表数据
3. URL包含 `/records` → 推断关联 `records` 表
4. 通过知识图谱查询：
   - `records` 表通过外键关联 `users` 表
   - `records` 表通过外键关联 `devices` 表
5. 系统推断接口依赖：
   - 需要先调用登录接口获取token
   - 需要先有用户数据（users表）
   - 需要先绑定设备（devices表）
   - 然后才能查询运动记录（records表）

### 示例3：创建订单接口

**接口信息**：
- URL: `POST /api/orders`
- 请求体: `{"user_id": 123, "product_id": 456, "quantity": 2}`

**关联过程**：
1. URL包含 `/orders` → 推断关联 `orders` 表
2. 参数包含 `user_id` → 推断需要 `users` 表数据
3. 参数包含 `product_id` → 推断需要 `products` 表数据
4. 通过知识图谱查询外键关系：
   - `orders.user_id` → `users.id`
   - `orders.product_id` → `products.id`
5. 生成测试数据时：
   - 从 `users` 表获取真实的用户ID
   - 从 `products` 表获取真实的产品ID
   - 生成符合 `orders` 表结构的订单数据

## 使用建议

1. **首次使用**：
   - 先创建数据库连接
   - 点击"分析数据库"按钮，提取数据库元数据
   - 等待分析完成（可能需要几分钟）

2. **接口上传后**：
   - 系统会自动分析接口参数
   - 在"接口依赖分析"页面可以看到接口之间的依赖关系
   - 系统会推断接口需要哪些表的数据

3. **生成测试数据时**：
   - 系统会根据接口信息和数据库元数据自动生成测试数据
   - 如果有外键约束，系统会从关联表获取真实数据

4. **提高准确性**：
   - 在数据库中添加表注释和字段注释，有助于系统更好地理解业务语义
   - 使用规范的表名和字段名（如使用下划线命名：user_id而不是userId）
   - 正确设置外键约束，系统可以通过外键关系自动推断表关联
