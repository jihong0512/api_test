## 系统概述

API接口智能测试系统是一个基于**大模型Agent**的智能API测试平台，采用**前后端分离**架构，利用AI能力自动化测试用例生成和执行。

### 系统目标
- 🤖 **AI驱动**：基于DeepSeek/通义千问大模型自动生成高质量测试用例
- 📚 **多格式支持**：支持OpenAPI/Swagger、JSON、YAML、PDF、Word等多种接口文档格式
- 🔗 **智能依赖分析**：自动分析接口间的依赖关系和调用链路
- 📊 **知识图谱**：使用Neo4j构建和可视化接口关系网络
- 🎯 **端到端测试**：从文档解析→接口识别→测试用例生成→用例执行→报告生成的完整流程

---

## 🤖 5个智能体概览

系统核心由**5个专业的智能体**协作完成，每个智能体专注一个核心任务：

### 智能体1: 文档解析智能体
**职责**：将各种格式文档转换为统一的接口清单
```
输入：上传的文件（PDF、Word、JSON、YAML等）
处理：识别格式 → 提取内容 → LLM标准化
输出：结构化接口列表 + 向量化存储
关键技术：多格式解析库 + DeepSeek LLM标准化
```

### 智能体2: 依赖分析智能体  
**职责**：分析接口之间的依赖关系，构建依赖图
```
输入：接口列表
处理：规则分析（CRUD顺序）→ LLM补充（隐含依赖）→ Neo4j存储
输出：依赖关系图 + 执行顺序
关键技术：Neo4j图数据库 + Kahn拓扑排序
```

### 智能体3: 测试用例生成智能体
**职责**：基于接口和依赖关系生成可执行的测试代码
```
输入：接口 + 依赖 + Few-shot + RAG上下文
处理：LLM理解 → 代码生成 → 格式验证
输出：Pytest代码文件
关键技术：Few-shot学习 + RAG向量检索 + LLM代码生成
```

### 智能体4: 数据准备与执行智能体
**职责**：为测试做准备，按正确顺序执行并自动传递数据
```
输入：测试用例 + 依赖图
处理：拓扑排序 → 数据方案规划 → 按序执行 → 数据提取传递
输出：测试执行结果 + Allure报告
关键技术：拓扑排序 + 异步执行 + 数据自动流转
```

### 智能体5: 错误分析与报告智能体
**职责**：分析测试失败原因，生成智能报告
```
输入：测试执行日志 + 错误信息
处理：LLM分析失败原因 → 生成建议
输出：详细的HTML/Allure报告
关键技术：错误分析LLM + 报告生成
```

**5个智能体的协作流程**：
```
用户上传文档
     ↓
[智能体1] 文档解析 → 接口清单
     ↓
[智能体2] 依赖分析 → Neo4j关系图
     ↓
[智能体3] 用例生成 → Pytest代码
     ↓
[智能体4] 数据准备执行 → Allure报告
     ↓
[智能体5] 错误分析 → 优化建议
```

## 核心特性

### 1. **智能文档解析**
```
文档上传 → 格式识别 → 内容提取 → AI理解 → 标准化
   ↓
支持格式: Swagger/OpenAPI、JSON、YAML、PDF、Word、Excel、TXT、Markdown
```

**核心能力**：
- 自动识别接口信息（名称、方法、URL、参数、响应）
- 从非结构化文档中提取结构化接口定义
- 支持多语言文档解析

### 2. **接口智能识别**
```
解析结果 → 接口提取 → 去重 → 验证 → 数据库存储
```

**包含信息**：
- 接口基本信息：名称、HTTP方法、URL、路径
- 请求信息：请求头、查询参数、请求体
- 响应信息：响应头、响应体、响应Schema
- 业务属性：描述、标签、版本、服务名

### 3. **接口依赖分析**
```
所有接口 → AI分析 → 依赖关系图 → 调用链路识别 → 执行顺序规划
```

**分析内容**：
- 接口间的数据依赖（如前置接口的返回值作为后续接口的输入）
- 业务流程链路（登录→获取权限→操作资源的流程）
- 参数传递关系

### 4. **AI驱动的测试用例生成**
```
接口信息 + 依赖关系 + 上下文 → AI生成 → 测试用例
```

**生成内容**：
- Pytest格式的Python测试代码
- 自动构造测试数据
- 设置前置条件（登录、创建资源等）
- 定义断言规则
- 处理响应数据提取

### 5. **RAG增强的知识检索**
```
用户查询 → 向量检索 + BM25 → 结果重排序 → 返回相关上下文
```

**应用场景**：
- 为LLM提供相关的接口文档上下文
- 支持语义搜索（"如何登录"→找到登录接口）
- 支持相似接口发现

## 系统组件

### 核心组件说明

#### 1. **前端服务** (React)
```
主要功能：
- 项目管理：创建、编辑、删除项目
- 文档管理：上传、查看、删除API文档
- 接口浏览：查看已解析的接口列表
- 用例管理：浏览、编辑、运行测试用例
- 报告查看：查看测试执行报告
```

#### 2. **后端服务** (FastAPI)
```
职责：
- 处理HTTP请求
- 调用各业务服务
- 与数据库交互
- 返回JSON响应
```

#### 3. **关键业务服务**

**DocumentParser（文档解析）**
```python
输入：上传的文档文件
过程：
  1. 识别文件格式
  2. 提取原始内容
  3. 使用LLM进行AI理解
  4. 标准化为统一格式
输出：结构化接口信息列表
```

**TestCaseGenerator（测试用例生成器）**
```python
输入：接口信息 + 测试数据 + 依赖关系
过程：
  1. 分析接口参数类型
  2. 生成测试数据
  3. 构造HTTP请求
  4. 使用LLM生成测试代码
  5. 添加断言和验证
输出：可执行的Pytest测试代码
```

**DependencyAnalyzer（依赖分析）**
```python
输入：所有解析的接口列表
过程：
  1. 识别接口间的参数依赖
  2. 识别业务流程链路
  3. 建立调用关系图
输出：依赖关系描述和执行顺序
```

**RAGService（RAG知识检索）**
```python
输入：用户查询/问题描述
过程：
  1. 向量化用户查询
  2. 从向量数据库检索相似文档块
  3. 使用BM25进行关键词匹配
  4. 对结果进行重排序
输出：最相关的接口文档片段
```

#### 4. **支撑系统**

**数据库 (MySQL)**
```
存储内容：
- 用户账户信息
- 项目元数据
- API接口定义
- 解析的接口详情
- 生成的测试用例
- 测试执行结果
```

**向量数据库 (Milvus/ChromaDB)**
```
存储内容：
- 文档块的向量表示
- 接口描述的向量表示
用途：
- 支持语义搜索
- 相似接口发现
```

**知识图谱 (Neo4j)**
```
存储关系：
- 接口→接口（依赖关系）
- 接口→数据库表（字段映射）
- 接口→业务实体（领域模型）
```

## 核心数据模型

### 1. **User（用户）**
```python
id: int (主键)
username: str (唯一)
email: str (唯一)
password_hash: str
created_at: datetime
updated_at: datetime
```

### 2. **Project（项目）**
```python
id: int (主键)
name: str
description: str
user_id: int (外键→User)
created_at: datetime
updated_at: datetime
```

### 3. **Document（文档）**
```python
id: int (主键)
project_id: int (外键→Project)
filename: str
file_type: str (json, yaml, pdf, docx等)
file_path: str (MinIO存储路径)
status: str (uploaded, parsing, completed, failed)
parse_result: str (解析后的JSON数据)
created_at: datetime
```

### 4. **DocumentAPIInterface（接口详情）**
```python
id: int (主键)
document_id: int (外键→Document)
project_id: int (外键→Project)

# 接口基本信息
name: str (接口名称)
method: str (GET, POST, PUT, DELETE)
url: str (完整URL或相对路径)
service: str (服务名，如user.userLogin)

# 请求信息
headers: json (请求头)
params: json (查询参数)
request_body: json (请求体Schema)

# 响应信息
response_body: json (响应体Schema)
response_schema: json (响应数据结构)
status_code: int (HTTP状态码)

# 其他属性
description: str
tags: json (标签数组)
version: str (API版本)
deprecated: bool

created_at: datetime
```

### 5. **TestCase（测试用例）**
```python
id: int (主键)
project_id: int (外键→Project)
api_interface_id: int (外键→APIInterface)
name: str (用例名称)
case_type: str (pytest or jmeter)
module: str (模块分类)
description: str

# 用例内容
test_data: json (测试数据)
test_code: str (生成的Python代码)
assertions: json (断言规则列表)
dependencies: str (依赖的前置条件)

# 生成追踪
status: str (active, generating, completed, failed)
generation_task_id: str (Celery任务ID)
generation_progress: int (0-100)
error_message: str

created_at: datetime
```

                              ├─→ Document (1) ──→ DocumentAPIInterface (多)
                              │

## 核心流程概览

### 工作流程
```
1️⃣ 上传文档
   用户上传API文档(Swagger/OpenAPI/PDF/Word等)
   │
2️⃣ 解析文档
   DocumentParser识别格式，提取接口信息
   LLM进行智能理解和标准化
   │
3️⃣ 存储接口
   接口信息存入DocumentAPIInterface表
   文档块向量化存入向量数据库
   │
4️⃣ 依赖分析
   DependencyAnalyzer分析接口间的依赖关系
   识别调用链路和参数流转
   │
5️⃣ 生成用例
   用户选择接口并触发生成
   TestCaseGenerator基于接口信息和依赖关系生成用例
   │
6️⃣ 执行测试
   运行生成的测试用例
   收集执行结果
   │
7️⃣ 生成报告
   汇总测试结果
   生成测试报告
```
