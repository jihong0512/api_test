## 数据模型概览

系统的数据模型采用**关系型设计**，核心有5个表：

```
┌──────────┐
│   User   │ (用户)
└────┬─────┘
     │
     ├─────────────┐
     │             │
     ▼             ▼
┌─────────┐  ┌──────────┐
│ Project │  │ Document │
└────┬────┘  └────┬─────┘
     │            │
     │            ▼
     │  ┌──────────────────────┐
     │  │ DocumentAPIInterface │ (接口详情)
     │  └──────────┬───────────┘
     │             │
     ▼             ▼
┌──────────┐
│ TestCase │
└──────────┘
```

**设计原则**
- ✅ 单一职责：每个表存储一类数据
- ✅ 避免冗余：相关数据通过外键关联
- ✅ 可扩展性：容易添加新的属性
- ✅ 查询效率：关键字段添加了索引

## User（用户）

### 表结构

```python
class User(Base):
    __tablename__ = "users"
    
    # 主键
    id = Column(Integer, primary_key=True, autoincrement=True)
    
    # 用户信息
    username = Column(String(255), unique=True, nullable=False, index=True)
    email = Column(String(255), unique=True, nullable=False, index=True)
    password_hash = Column(String(255), nullable=False)
    
    # 时间戳
    created_at = Column(DateTime, default=datetime.utcnow)
    updated_at = Column(DateTime, default=datetime.utcnow, onupdate=datetime.utcnow)
    
    # 关系
    projects = relationship("Project", back_populates="user")
```

### 字段说明

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| `id` | INT | PK | 主键，自增 |
| `username` | VARCHAR(255) | UNIQUE, NOT NULL | 唯一的用户名 |
| `email` | VARCHAR(255) | UNIQUE, NOT NULL | 唯一的邮箱 |
| `password_hash` | VARCHAR(255) | NOT NULL | 密码的哈希值（不存储明文）|
| `created_at` | DATETIME | DEFAULT NOW | 创建时间 |
| `updated_at` | DATETIME | DEFAULT NOW | 最后更新时间 |

### 创建用户示例

```python
# 注册新用户
user = User(
    username="zhangsan",
    email="zhangsan@example.com",
    password_hash=hash_password("password123")  # 密码必须哈希
)
db.session.add(user)
db.session.commit()

# 登录验证
user = db.query(User).filter(User.username == "zhangsan").first()
if user and verify_password(user.password_hash, input_password):
    # 登录成功
    pass
```

### 索引

```
理由：
  - username和email是登录时的查询条件
  - 频繁的查询需要索引提高速度
  
索引设置：
  CREATE UNIQUE INDEX idx_username ON users(username);
  CREATE UNIQUE INDEX idx_email ON users(email);
```

## Project（项目）

### 表结构

```python
class Project(Base):
    __tablename__ = "projects"
    
    # 主键
    id = Column(Integer, primary_key=True, autoincrement=True)
    
    # 项目信息
    name = Column(String(255), nullable=False)
    description = Column(Text, default="")
    
    # 关键外键
    user_id = Column(Integer, ForeignKey("users.id"), nullable=False, index=True)
    
    # 时间戳
    created_at = Column(DateTime, default=datetime.utcnow)
    updated_at = Column(DateTime, default=datetime.utcnow, onupdate=datetime.utcnow)
    
    # 关系
    user = relationship("User", back_populates="projects")
    documents = relationship("Document", back_populates="project")
    test_cases = relationship("TestCase", back_populates="project")
```

### 字段说明

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| `id` | INT | PK | 主键，自增 |
| `name` | VARCHAR(255) | NOT NULL | 项目名称（同一用户下可重名） |
| `description` | TEXT | DEFAULT '' | 项目描述 |
| `user_id` | INT | FK→users.id | 项目所有者 |
| `created_at` | DATETIME | DEFAULT NOW | 创建时间 |
| `updated_at` | DATETIME | DEFAULT NOW | 最后更新时间 |

### 创建项目示例

```python
# 用户创建项目
project = Project(
    name="电商API",
    description="包含订单、商品、支付等接口",
    user_id=user.id  # 关联用户
)
db.session.add(project)
db.session.commit()

# 查询用户的所有项目
projects = db.query(Project).filter(Project.user_id == user.id).all()

# 删除项目时会级联删除相关文档和测试用例
db.session.delete(project)
db.session.commit()
```

### 级联关系

```
当删除项目时：
  Project deleted
    ↓ (级联删除)
    ├─ Document(s) deleted
    │   ├─ DocumentAPIInterface(s) deleted
    │   └─ TestCase(s) deleted
    └─ TestCase(s) deleted

级联删除的作用：
  - 保持数据一致性（没有孤立的文档或用例）
  - 简化删除逻辑（不用手动删除相关数据）
```

## Document（文档）

### 表结构

```python
class Document(Base):
    __tablename__ = "documents"
    
    # 主键
    id = Column(Integer, primary_key=True, autoincrement=True)
    
    # 关键外键
    project_id = Column(Integer, ForeignKey("projects.id"), nullable=False, index=True)
    
    # 文档信息
    filename = Column(String(255), nullable=False)
    file_type = Column(String(50), nullable=False)  # json, yaml, pdf, docx等
    file_path = Column(String(500), nullable=False)  # MinIO存储路径或本地路径
    
    # 解析状态
    status = Column(String(50), default="uploaded", nullable=False)
    # 可能的值：uploaded, parsing, completed, failed
    
    # 解析结果
    parse_result = Column(JSON, default={})  # 存储解析后的原始数据
    error_message = Column(Text, default="")  # 如果解析失败，存储错误信息
    
    # 时间戳
    created_at = Column(DateTime, default=datetime.utcnow)
    updated_at = Column(DateTime, default=datetime.utcnow, onupdate=datetime.utcnow)
    
    # 关系
    project = relationship("Project", back_populates="documents")
    api_interfaces = relationship("DocumentAPIInterface", back_populates="document")
```

### 字段说明

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | INT | 主键 |
| `project_id` | INT | 所属项目（外键）|
| `filename` | VARCHAR(255) | 原始文件名（如 "Swagger.json"）|
| `file_type` | VARCHAR(50) | 文件格式（json、yaml、pdf、docx等）|
| `file_path` | VARCHAR(500) | 文件存储路径（MinIO或本地）|
| `status` | VARCHAR(50) | 解析状态（uploaded/parsing/completed/failed）|
| `parse_result` | JSON | 解析后的完整数据 |
| `error_message` | TEXT | 失败时的错误信息 |
| `created_at` | DATETIME | 上传时间 |
| `updated_at` | DATETIME | 最后更新时间 |

### 文件上传和解析流程

```python
# 1. 用户上传文件
document = Document(
    project_id=project.id,
    filename="swagger.json",
    file_type="json",
    file_path="projects/1/swagger.json",  # MinIO路径
    status="uploaded"
)
db.session.add(document)
db.session.commit()

# 2. 后台异步启动解析任务（Celery）
parse_document_task.delay(document.id)

# 3. 解析任务更新状态
document.status = "parsing"
db.session.commit()

# 4. 解析完成，更新parse_result
document.status = "completed"
document.parse_result = {
    "swagger": "2.0",
    "info": {...},
    "paths": {...}
}
db.session.commit()

# 5. 或者解析失败
document.status = "failed"
document.error_message = "Unsupported file format or corrupted file"
db.session.commit()
```

### 支持的文件格式

| 格式 | file_type | 特点 | 示例 |
|------|-----------|------|------|
| **OpenAPI** | openapi | 标准的API描述格式（3.0版本）| openapi.yaml |
| **Swagger** | swagger | OpenAPI的前身（2.0版本）| swagger.json |
| **JSON** | json | 自定义的JSON格式 API文档 | api.json |
| **YAML** | yaml | 自定义的YAML格式 API文档 | api.yaml |
| **PDF** | pdf | 扫描或导出的PDF文档 | API文档.pdf |
| **Word** | docx | Microsoft Word文档 | API文档.docx |
| **Excel** | xlsx | 电子表格格式 | API列表.xlsx |
| **Markdown** | md | Markdown格式的API文档 | README.md |
| **Text** | txt | 纯文本格式 | api_doc.txt |
| **HTML** | html | 网页格式 | api.html |

## DocumentAPIInterface（接口详情）

### 表结构

这是**最复杂的表**，存储解析后的接口详细信息。

```python
class DocumentAPIInterface(Base):
    __tablename__ = "document_api_interfaces"
    
    # 主键
    id = Column(Integer, primary_key=True, autoincrement=True)
    
    # 关键外键
    document_id = Column(Integer, ForeignKey("documents.id"), nullable=False, index=True)
    project_id = Column(Integer, ForeignKey("projects.id"), nullable=False, index=True)
    
    # ===== 接口基本信息 =====
    name = Column(String(255), nullable=False)  # 接口名称
    method = Column(String(10), nullable=False)  # GET, POST, PUT, DELETE等
    url = Column(String(500), nullable=False)  # URL路径
    service = Column(String(255), default="")  # 服务名，如 "user.login"
    
    # ===== 请求定义 =====
    headers = Column(JSON, default={})  # 请求头Schema
    params = Column(JSON, default={})  # 查询参数Schema
    request_body = Column(JSON, default={})  # 请求体Schema
    
    # ===== 响应定义 =====
    response_body = Column(JSON, default={})  # 响应体Schema
    response_schema = Column(JSON, default={})  # 详细的响应数据结构
    status_code = Column(Integer, default=200)  # 成功状态码
    error_responses = Column(JSON, default={})  # 错误响应示例 {400: {...}, 401: {...}}
    
    # ===== 元数据 =====
    description = Column(Text, default="")  # 接口描述
    tags = Column(JSON, default=[])  # 标签，如 ["用户管理", "认证"]
    version = Column(String(50), default="1.0")  # API版本
    deprecated = Column(Boolean, default=False)  # 是否已废弃
    
    # ===== 时间戳 =====
    created_at = Column(DateTime, default=datetime.utcnow)
    updated_at = Column(DateTime, default=datetime.utcnow, onupdate=datetime.utcnow)
    
    # ===== 关系 =====
    document = relationship("Document", back_populates="api_interfaces")
    test_cases = relationship("TestCase", back_populates="api_interface")
```

### 字段详解

#### 基本字段

```python
name: str       # "用户登录"
method: str     # "POST"
url: str        # "/user/login" 或 "https://api.example.com/user/login"

```

## TestCase（测试用例）

### 表结构

```python
class TestCase(Base):
    __tablename__ = "test_cases"
    
    # 主键
    id = Column(Integer, primary_key=True, autoincrement=True)
    
    # 关键外键
    project_id = Column(Integer, ForeignKey("projects.id"), nullable=False, index=True)
    api_interface_id = Column(Integer, ForeignKey("document_api_interfaces.id"), nullable=False, index=True)
    
    # ===== 基本信息 =====
    name = Column(String(255), nullable=False)  # 用例名称
    case_type = Column(String(50), default="pytest")  # pytest 或 jmeter
    module = Column(String(255), default="")  # 模块分类，如 "用户管理/登录"
    description = Column(Text, default="")
    
    # ===== 用例内容 =====
    test_data = Column(JSON, default={})  # 测试数据
    test_code = Column(Text, default="")  # 生成的Python或JMeter代码
    assertions = Column(JSON, default=[])  # 断言规则列表
    dependencies = Column(JSON, default=[])  # 依赖的前置用例或接口
    
    # ===== 生成进度 =====
    status = Column(String(50), default="draft")
    # 可能的值：draft, generating, completed, error, deprecated
    
    generation_task_id = Column(String(255), default="")  # Celery任务ID
    generation_progress = Column(Integer, default=0)  # 进度 0-100
    error_message = Column(Text, default="")  # 生成失败时的错误
    
    # ===== 时间戳 =====
    created_at = Column(DateTime, default=datetime.utcnow)
    updated_at = Column(DateTime, default=datetime.utcnow, onupdate=datetime.utcnow)
    
    # ===== 关系 =====
    project = relationship("Project", back_populates="test_cases")
    api_interface = relationship("DocumentAPIInterface", back_populates="test_cases")
```

### 字段说明

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | INT | 主键 |
| `project_id` | INT | 所属项目 |
| `api_interface_id` | INT | 所属接口 |
| `name` | VARCHAR(255) | 用例名称 |
| `case_type` | VARCHAR(50) | pytest/jmeter |
| `module` | VARCHAR(255) | 模块分类 |
| `test_data` | JSON | 测试数据 |
| `test_code` | TEXT | 生成的测试代码 |
| `assertions` | JSON | 断言规则 |
| `dependencies` | JSON | 前置条件 |
| `status` | VARCHAR(50) | 用例状态 |

### 用例生成流程

```python
# 1. 用户触发生成
test_case = TestCase(
    project_id=1,
    api_interface_id=10,
    name="正常登录流程测试",
    case_type="pytest",
    module="用户管理",
    status="generating",
    generation_task_id="task_xyz123"
)
db.session.add(test_case)
db.session.commit()

# 2. 后台生成（Celery任务）
# - 调用LLM生成测试代码
# - 周期更新generation_progress

# 3. 生成完成
test_case.test_code = """
def test_user_login():
    response = requests.post(
        url="https://api.example.com/user/login",
        json={"username": "test_user", "password": "password123"}
    )
    assert response.status_code == 200
    assert "token" in response.json()["data"]
"""
test_case.status = "completed"
test_case.test_data = {
    "username": "test_user",
    "password": "password123"
}
test_case.assertions = [
    {"field": "status_code", "operator": "==", "expected": 200},
    {"field": "data.token", "operator": "exists"}
]
db.session.commit()

# 4. 执行用例时
# - 使用test_code执行测试
# - 记录结果和日志
```

### 用例状态流转

```
创建
  ↓
[draft] 草稿状态
  ↓ (用户点击生成)
[generating] 生成中
  ├─ 成功 → [completed] 完成
  └─ 失败 → [error] 错误
  
[completed] 或 [error]
  ↓ (用户停用用例)
[deprecated] 已废弃
```

## 模型关系

### 关系图

```
User (1) ──┬─→ Project (多) ──┬─→ Document (多) ──→ DocumentAPIInterface (多)
           │                  │                            ↓
           └──────────────────┴─→ TestCase (多) ←──────────┘
```

### 关系说明

#### 1. User ↔ Project （一对多）

```python
# 一个用户可以有多个项目
user.projects  # 返回该用户的所有项目

# 一个项目属于一个用户
project.user   # 返回该项目的所有者
```

#### 2. Project ↔ Document （一对多）

```python
# 一个项目可以上传多个文档（如多个Swagger文件）
project.documents  # 返回该项目的所有文档

# 一个文档属于一个项目
document.project   # 返回该文档所属的项目
```

#### 3. Document ↔ DocumentAPIInterface （一对多）

```python
# 一个文档可以包含多个接口
document.api_interfaces  # 返回该文档中识别的所有接口

# 一个接口来自一个文档
api_interface.document   # 返回该接口所来自的文档
```

#### 4. DocumentAPIInterface ↔ TestCase （一对多）

```python
# 一个接口可以有多个测试用例（不同的场景）
api_interface.test_cases  # 返回该接口的所有测试用例

# 一个测试用例对应一个接口
test_case.api_interface   # 返回该用例测试的接口
```

#### 5. Project ↔ TestCase （一对多）

```python
# 一个项目可以有多个测试用例（来自不同的接口）
project.test_cases  # 返回该项目的所有测试用例

# 一个测试用例属于一个项目
test_case.project   # 返回该用例所属的项目
```

### 外键约束

```sql
ALTER TABLE projects ADD FOREIGN KEY (user_id) REFERENCES users(id) ON DELETE CASCADE;
ALTER TABLE documents ADD FOREIGN KEY (project_id) REFERENCES projects(id) ON DELETE CASCADE;
ALTER TABLE document_api_interfaces ADD FOREIGN KEY (document_id) REFERENCES documents(id) ON DELETE CASCADE;
ALTER TABLE document_api_interfaces ADD FOREIGN KEY (project_id) REFERENCES projects(id) ON DELETE CASCADE;
ALTER TABLE test_cases ADD FOREIGN KEY (project_id) REFERENCES projects(id) ON DELETE CASCADE;
ALTER TABLE test_cases ADD FOREIGN KEY (api_interface_id) REFERENCES document_api_interfaces(id) ON DELETE CASCADE;
```

**级联删除的含义**：
- 当删除一个项目时，该项目下的所有文档、接口、用例都会被删除
- 这确保了数据的完整性和一致性

## 数据流转示例

### 完整的数据生命周期

```
1️⃣ 用户注册和创建项目
   ├─ 创建 User 记录
   └─ 创建 Project 记录（关联 user_id）

2️⃣ 上传API文档
   ├─ 创建 Document 记录（状态：uploaded）
   └─ 文件存储到 MinIO

3️⃣ 解析文档（后台异步）
   ├─ Document.status: uploading → parsing
   └─ DocumentAPIInterface（多条记录，每个接口一条）

4️⃣ 生成测试用例（后台异步）
   ├─ 遍历所有 DocumentAPIInterface
   ├─ 调用LLM生成每个接口的 TestCase
   └─ 保存 test_code

5️⃣ 执行测试
   ├─ 读取 TestCase.test_code
   ├─ 按照依赖关系执行
   └─ 保存执行结果

6️⃣ 生成报告
```

## 总结

| 模型 | 用途 | 关键字段 | 关系 |
|------|------|--------|------|
| **User** | 用户认证 | username, email, password_hash | 1 对多 Project |
| **Project** | 项目分组 | name, user_id | 1 对多 Document, TestCase |
| **Document** | 文档管理 | filename, file_path, status, parse_result | 1 对多 APIInterface |
| **DocumentAPIInterface** | 接口存储 | name, method, url, request_body, response_body | 1 对多 TestCase |
| **TestCase** | 测试用例 | test_code, test_data, assertions, status | 多对1 APIInterface |

通过这5个表的有机结合，系统实现了**从上传文档到自动生成和执行测试的完整数据流**！
