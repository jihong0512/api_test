## 依赖分析概述

### 核心目标
```
找出接口之间的依赖关系，规划正确的执行顺序
```

### 为什么需要依赖分析？

**问题场景**：
```
假设有这样的接口：

1. 登录         POST /login
   返回：token, userId

2. 创建文章     POST /articles
   需要：token, userId

3. 获取文章     GET /articles/{id}
   需要：articleId

4. 更新文章     PUT /articles/{id}
   需要：token, userId, articleId

5. 删除文章     DELETE /articles/{id}
   需要：token, articleId
```

**问题**：
- ❌ 如果直接测试接口2，会失败（缺少token）
- ❌ 如果直接测试接口3，会失败（缺少articleId）
- ✅ 正确的执行顺序应该是：1 → 2 → 3 → 4 → 5

**分析的作用**：
- 自动发现接口间的数据流转关系
- 规划测试执行顺序（保证前置条件满足）
- 自动化构造测试数据（从前置接口的响应中提取）
- 识别业务逻辑流程

## 依赖类型

### 1. **参数依赖**（数据流依赖）

当一个接口的**输入参数**来自另一个接口的**响应数据**时

**识别方式**：
```
接口A的响应字段 = 接口B的请求参数
```

**示例**：
```
接口A (登录):
请求: {username, password}
响应: {code: 0, data: {token: "abc123", userId: 123}}

接口B (获取用户信息):
请求: {userId: xxx, Authorization: "Bearer {token}"}
响应: {code: 0, data: {id, name, email...}}

分析结果：
接口B 依赖于 接口A
- 接口A.response.data.userId → 接口B.request.userId
- 接口A.response.data.token → 接口B.request.headers.Authorization
```

**常见的参数名对应**：
```
userId          → 接口B.userId 或 {id}
token/accessToken → Authorization 或 headers.token
id/objectId     → 路径参数 /{id}
orderId         → 接口C.orderId
```

### 2. **业务流程依赖**（顺序依赖）

当多个接口组成一个完整的业务流程时

**示例 - 电商流程**：
```
1. 登录
   POST /auth/login

2. 查看商品
   GET /products

3. 添加到购物车
   POST /cart/items

4. 查看购物车
   GET /cart

5. 创建订单
   POST /orders

6. 支付订单
   POST /orders/{orderId}/payment

7. 查询订单状态
   GET /orders/{orderId}

执行顺序依赖：
1 → 2（登录后才能浏览）
1 → 3（需要token）
3 → 4（添加后才能查看）
4 → 5（生成订单需要购物车信息）
5 → 6（支付需要订单）
6 → 7（支付后查询）
```

### 3. **资源依赖**（前置创建）

当接口操作的资源需要先被创建时

**示例**：
```
获取用户      GET /users/{userId}
├─ 依赖于 → 创建用户   POST /users
           响应返回 userId

更新用户      PUT /users/{userId}
├─ 依赖于 → 创建用户   POST /users
           响应返回 userId

删除用户      DELETE /users/{userId}
├─ 依赖于 → 创建用户   POST /users
           响应返回 userId
```

### 4. **认证依赖**（权限）

接口需要有效的认证令牌

**示例**：
```
大多数接口需要：
- Authorization: Bearer {token}
- 来自登录接口的响应

所以流程应该是：
登录 → 其他所有需要auth的接口
```

## 分析方法

### 方法1：**参数名称匹配**（简单启发式）

最基础的分析方法

```python
def analyze_parameter_dependencies(interfaces: List[dict]):
    """
    通过参数名称匹配识别依赖
    """
    dependencies = []
    
    for target_api in interfaces:
        # 收集target_api的所有输入参数名
        target_params = extract_all_parameters(target_api)
        
        for source_api in interfaces:
            if source_api == target_api:
                continue
            
            # 收集source_api的所有输出字段名
            source_fields = extract_all_response_fields(source_api)
            
            # 检查是否有匹配
            matching_fields = target_params & source_fields
            if matching_fields:
                dependencies.append({
                    "source": source_api["name"],
                    "target": target_api["name"],
                    "mapped_fields": list(matching_fields),
                    "confidence": "high" if len(matching_fields) >= 2 else "medium"
                })
    
    return dependencies

# 示例
interfaces = [
    {
        "name": "登录",
        "method": "POST",
        "url": "/api/login",
        "response": {"token": "xxx", "userId": 123}
    },
    {
        "name": "获取用户",
        "method": "GET",
        "url": "/api/users/{userId}",
        "headers": {"Authorization": "Bearer {token}"}
    }
]

deps = analyze_parameter_dependencies(interfaces)
# 结果：获取用户 依赖于 登录（共享token, userId）
```

**优点**：快速，直观
**缺点**：可能误判（参数名称相同但无关）

## Neo4j：为什么用它存储依赖关系？

### 问题场景：为什么不用MySQL？

**需要查询的场景**：
```
查询：接口C需要哪些前置接口？
     更深一层：这些前置接口又依赖谁？
     再深一层：完整的依赖链是什么？

用 MySQL 查询很复杂：
SELECT * FROM api_dependencies 
WHERE target_id = 'C'   -- 获取直接依赖
UNION
SELECT * FROM api_dependencies d1
JOIN api_dependencies d2 ON d1.source_id = d2.target_id
WHERE d2.target_id = 'C'  -- 获取二阶依赖
UNION ...  -- 继续嵌套（太复杂了！）

用 Neo4j 查询很简单：
MATCH (C:APIInterface)<-[:DEPENDS_ON*]-(ancestor:APIInterface)
RETURN ancestor  -- 一行代码！
```

### Neo4j的优势

| 优势 | 说明 |
|------|------|
| **图查询简单** | 多跳依赖用Cypher只需一行代码 |
| **性能好** | Neo4j专门优化了图遍历算法 |
| **可视化** | 自带可视化工具，直观展示关系 |
| **支持复杂查询** | 路径查询、最短路径、社区检测等 |

### 使用Neo4j存储接口关系

Neo4j是图数据库，特别适合存储接口依赖关系

```cypher
# 创建节点和关系

# 1. 创建接口节点
CREATE (n:APIInterface {
    id: "api_1",
    name: "登录",
    method: "POST",
    url: "/api/login",
    projectId: 123
})

# 2. 创建依赖关系
CREATE (api_1)-[:DEPENDS_ON {
    type: "parameter",
    fields: ["userId", "token"],
    confidence: 0.95
}]->(api_2)

# 3. 查询依赖
MATCH (a:APIInterface)-[:DEPENDS_ON]->(b:APIInterface)
WHERE a.projectId = 123
RETURN a.name, b.name
```

### 关系图的应用

#### **应用1：执行顺序规划**
```cypher
# 查询接口的所有直接和间接依赖
MATCH path = (target:APIInterface)-[:DEPENDS_ON*]->(source:APIInterface)
WHERE target.id = "api_3" AND target.projectId = 123
RETURN path
ORDER BY length(path)
```

#### **应用2：识别业务流程**
```cypher
# 找到最长的依赖链（业务流程）
MATCH path = (a:APIInterface)-[:DEPENDS_ON*]->(b:APIInterface)
WHERE a.projectId = 123
RETURN path, length(path) as depth
ORDER BY depth DESC
LIMIT 1
```

#### **应用3：找出关键接口**
```cypher
# 找到被最多接口依赖的接口（关键）
MATCH (a:APIInterface)-[:DEPENDS_ON]->(critical:APIInterface)
WHERE critical.projectId = 123
RETURN critical.name, count(*) as dependency_count
ORDER BY dependency_count DESC
```

## 拓扑排序：把"图"变成"执行顺序"

### 什么是拓扑排序？

**生活例子**：
```
想象你要做饭：

1. 买菜（无依赖）
2. 洗菜（依赖买菜）
3. 切菜（依赖洗菜）
4. 炒菜（依赖切菜）
5. 装盘（依赖炒菜）

问题：这5个步骤怎么有序执行？
答案：拓扑排序 = [买菜, 洗菜, 切菜, 炒菜, 装盘]
```

**在接口测试中**：
```
3个测试用例：

1. 登录（无依赖）
2. 创建订单（依赖登录获得token）
3. 查询订单（依赖创建订单获得order_id）

拓扑排序结果 = [登录, 创建订单, 查询订单]
```

### Kahn算法：最常用的拓扑排序算法

**核心思想**：
```
1. 找出所有"没有前置任务"的任务（入度为0）
2. 执行它们，删除它们的出度
3. 重复1-2，直到所有任务都完成
4. 如果无法完成，说明有循环依赖
```

**什么是"入度"？**
```
入度 = "有多少个其他任务依赖于我"

例子：
任务A（入度0）：没人等我的结果，可以立即开始
任务B（入度1）：有1个任务（A）依赖我，我要等A完成后才能开始
任务C（入度2）：有2个任务（A、B）依赖我
```

### Kahn算法的演示

**初始状态**：
```
任务A（入度0）← 可以开始
任务B（入度1）← 等A
任务C（入度1）← 等B
```

**执行过程**：

```
步骤1：执行A（入度为0）
  结果列表 = [A]
  删除A，更新B的入度：1-1=0

步骤2：执行B（入度变0）
  结果列表 = [A, B]
  删除B，更新C的入度：1-1=0

步骤3：执行C（入度变0）
  结果列表 = [A, B, C]
  删除C，队列空

最终顺序：[A, B, C] ✓
```

### 在apitest中的实现

**核心代码位置**：
```python
# backend/app/services/task_preparation.py
def _sort_test_cases_by_dependency(test_cases, dependency_graph):
    """拓扑排序：根据依赖关系排序测试用例"""
```

**完整实现逻辑**：

```python
def topological_sort(test_cases, dependencies):
    """
    Kahn 算法实现拓扑排序
    """
    # 1. 初始化入度表
    in_degree = {}
    for case in test_cases:
        in_degree[case.id] = 0
    
    # 2. 计算每个用例的入度（有多少个其他用例依赖它）
    for dep in dependencies:
        if dep['target_id'] not in in_degree:
            in_degree[dep['target_id']] = 0
        in_degree[dep['target_id']] += 1
    
    # 3. 初始化队列：入度为0的用例可以先执行
    queue = [case_id for case_id, degree in in_degree.items() if degree == 0]
    
    # 4. Kahn 算法的核心循环
    result = []
    while queue:
        # 取出一个入度为0的用例
        case_id = queue.pop(0)
        result.append(case_id)
        
        # 更新依赖于这个用例的其他用例
        for dep in dependencies:
            if dep['source_id'] == case_id:
                # source_id 就是 case_id 的所有依赖者
                target_id = dep['target_id']
                in_degree[target_id] -= 1
                
                # 如果入度变0，加入队列
                if in_degree[target_id] == 0:
                    queue.append(target_id)
    
    # 5. 检查是否有循环依赖
    if len(result) != len(test_cases):
        raise ValueError("存在循环依赖！无法完成排序")
    
    return result
```

### 拓扑排序的完整例子

**输入**：
```python
test_cases = [
    {"id": 1, "name": "登录"},
    {"id": 2, "name": "创建订单"},
    {"id": 3, "name": "查询订单"}
]

dependencies = [
    {"source_id": 1, "target_id": 2},  # 创建订单 → 登录
    {"source_id": 2, "target_id": 3}   # 查询订单 → 创建订单
]
```

**执行过程**：
```
初始化：
  in_degree = {1: 0, 2: 1, 3: 1}
  queue = [1]  （只有登录的入度为0）
  result = []

迭代1：
  pop 1, result = [1]
  找依赖于1的：2
  in_degree[2] = 1 - 1 = 0, queue = [2]

迭代2：
  pop 2, result = [1, 2]
  找依赖于2的：3
  in_degree[3] = 1 - 1 = 0, queue = [3]

迭代3：
  pop 3, result = [1, 2, 3]
  找依赖于3的：无
  queue = []

结束：result = [1, 2, 3]
```

**输出**：
```python
[1, 2, 3]  # 执行顺序：登录 → 创建订单 → 查询订单
```

### 拓扑排序的作用

✅ **确保依赖满足**：前置接口必定在后续接口之前执行
✅ **自动提取数据**：前面的响应结果自动提取，传递给后面的请求
✅ **检测错误**：发现循环依赖（接口A依赖B，B又依赖A的问题）
✅ **并行优化**（高级）：入度为0的用例可以并行执行

## 执行顺序规划

### 拓扑排序（Topological Sort）

根据依赖关系计算接口的执行顺序

```python
from collections import defaultdict, deque

def topological_sort(interfaces: List[dict], dependencies: List[dict]):
    """
    拓扑排序：确保依赖关系正确的执行顺序
    """
    # 构建图
    graph = defaultdict(list)  # api_name -> [依赖它的api]
    in_degree = defaultdict(int)  # api_name -> 依赖数
    
    # 初始化
    for api in interfaces:
        if api["name"] not in in_degree:
            in_degree[api["name"]] = 0
    
    # 添加边
    for dep in dependencies:
        graph[dep["source"]].append(dep["target"])
        in_degree[dep["target"]] += 1
    
    # 拓扑排序
    queue = deque([api["name"] for api in interfaces if in_degree[api["name"]] == 0])
    result = []
    
    while queue:
        api = queue.popleft()
        result.append(api)
        
        for dependent in graph[api]:
            in_degree[dependent] -= 1
            if in_degree[dependent] == 0:
                queue.append(dependent)
    
    if len(result) != len(interfaces):
        raise ValueError("存在循环依赖！")
    
    return result

# 示例
interfaces = [
    {"name": "登录"},
    {"name": "获取用户"},
    {"name": "更新用户"},
    {"name": "删除用户"}
]

dependencies = [
    {"source": "登录", "target": "获取用户"},
    {"source": "登录", "target": "更新用户"},
    {"source": "登录", "target": "删除用户"},
    {"source": "获取用户", "target": "更新用户"}
]

order = topological_sort(interfaces, dependencies)
# 结果: ['登录', '获取用户', '更新用户', '删除用户']
```

### 执行计划生成

def generate_execution_plan(

    in_degree = defaultdict(int)
    
    for api in interfaces:
        if api["name"] not in in_degree:
            in_degree[api["name"]] = 0
    
    for dep in dependencies:
        graph[dep["source"]].append(dep["target"])
        in_degree[dep["target"]] += 1
    
    # 分层执行
    plan = []
    remaining = set(api["name"] for api in interfaces)
    
    while remaining:
        # 找出当前可以执行的接口（无依赖或依赖已满足）
        current_level = [
            api for api in remaining
            if in_degree[api] == 0
        ]
        
        if not current_level:
            raise ValueError("存在循环依赖")
        
        plan.append(current_level)
        
        # 更新状态
        for api in current_level:
            remaining.remove(api)
            for dependent in graph[api]:
                in_degree[dependent] -= 1
    
    return plan

# 结果示例:
# [
#   ['登录'],                    # 第1批：只需登录
#   ['获取用户', '创建文章'],    # 第2批：可以并行执行
#   ['更新用户'],                # 第3批：需要获取用户的结果
#   ['删除用户']                 # 第4批：最后删除
# ]
```

class DependencyAnalyzer:
    """接口依赖分析器"""
    
    def __init__(self):
        self.llm_service = LLMService()
    
    async def analyze(self, interfaces: List[dict], project_id: int):
        """
        分析接口依赖
        """
        # 步骤1：参数名称匹配（快速）
        simple_deps = self._find_parameter_matches(interfaces)
        
        # 步骤2：LLM增强（补全业务流程等）
        enhanced_deps = await self._enhance_with_llm(
            interfaces, 
            simple_deps,
            project_id
        )
        
        # 步骤3：生成执行计划
        execution_plan = self._generate_execution_plan(interfaces, enhanced_deps)
        
        # 步骤4：存储到Neo4j
        await self._save_to_neo4j(interfaces, enhanced_deps, project_id)
        
        return {
            "dependencies": enhanced_deps,
            "execution_plan": execution_plan,
            "stats": {
                "total_interfaces": len(interfaces),
                "total_dependencies": len(enhanced_deps),
                "circular_dependencies": 0
            }
        }
    
    def _find_parameter_matches(self, interfaces: List[dict]):
        """简单的参数名称匹配"""
        deps = []
        
        for target_api in interfaces:
            target_params = self._extract_all_parameters(target_api)
            
            for source_api in interfaces:
                if source_api == target_api:
                    continue
                
                source_fields = self._extract_response_fields(source_api)
                matches = target_params & source_fields
                
                if matches:
                    deps.append({
                        "source": source_api["name"],
                        "target": target_api["name"],
                        "type": "parameter",
                        "fields": list(matches)
                    })
        
        return deps
    
    async def _enhance_with_llm(self, interfaces, simple_deps, project_id):
        """使用LLM增强分析结果"""
        prompt = f"""
        基于以下接口定义和初步分析，完整分析接口依赖：
        
        接口列表：{json.dumps(interfaces, ensure_ascii=False, indent=2)}
        
        初步依赖分析：{json.dumps(simple_deps, ensure_ascii=False, indent=2)}
        
        请补充：
        1. 业务流程依赖
        2. 隐含的认证依赖
        3. 资源创建依赖
        
        返回完整的JSON...
        """
        
        result = await self.llm_service.chat(prompt)
        return json.loads(result)
```

## 最佳实践

### ✅ 推荐做法

1. **先使用参数名称匹配快速分析**
2. **再用LLM验证和补全**
3. **存储到图数据库便于查询**
4. **可视化展示依赖关系**
5. **定期更新依赖信息**

### ⚠️ 常见陷阱

1. ❌ 忽视业务流程的隐含依赖
2. ❌ 参数名不同但语义相同的字段识别失败
3. ❌ 未处理循环依赖
4. ❌ 认证依赖处理不完整
