## 一、大白话：5个智能体的完整流程

### 1.1 核心概念：智能体（Agent）

什么是"智能体"？
- 不是某个单独的类或函数
- 而是一个"职能完整的模块"，包含规则 + LLM + 存储
- 每个智能体负责一个完整的业务环节

### 1.3 5大流程详解（从用户的角度）

#### 流程1：用户上传文档

**用户做什么？**
```
在前端上传一份接口文档（OpenAPI/Swagger、PDF、Word 等）
```

**系统做什么？**
```
文档解析智能体立即启动
```

#### 流程2：智能体1 解析文档 → 统一接口清单

**比喻**：
像一个"特别会看文档的测试同事"，把文档里所有接口都"抄出来"变成结构化数据。

**具体步骤**：
1. 识别文档格式（PDF、Word、JSON...）
2. 用对应库解析（pdfplumber、python-docx、yaml...）
3. 用 LLM（DeepSeek）标准化数据
4. 输出统一 Schema：

```json
{
  "interfaces": [
    {
      "name": "用户登录",
      "method": "POST",
      "url": "/api/login",
      "headers": {"Content-Type": "application/json"},
      "body": {"username": "", "password": ""},
      "response_schema": {"token": "string"}
    }
  ]
}
```

**为什么需要 LLM 标准化？**
- PDF 可能有表格，Word 可能有文字，JSON 可能嵌套很深
- LLM 能理解所有这些格式，统一转换成标准 Schema
- 后续智能体就只需要处理一种格式

#### 流程3：智能体2 依赖分析 → 依赖关系图

**比喻**：
像一个"架构师"，把接口之间的关系理清楚。

**谁依赖谁？**
```
创建订单 ← 用户登录（需要token）
查询订单 ← 创建订单（需要order_id）
```

**分析方法**：
1. **规则分析**（快速，不用LLM）：
   - 看接口名、路径、HTTP方法
   - 识别CRUD类型（创建→更新→查询→删除）
   - 按CRUD顺序自动连接

2. **LLM分析**（精准，补充规则的不足）：
   - 识别"需要token"、"需要XXX_id"这种隐含依赖
   - 理解业务逻辑依赖

3. **存储到 Neo4j**（图数据库）：
   - 节点 = 接口
   - 边 = 依赖关系
   - 优势：查询和可视化都很快

#### 流程4：智能体3 用例生成 → pytest 代码

**比喻**：
像一个"会写代码的高级测试"，把前面这些信息翻译成真正能跑的代码。

**输入数据**：
1. 接口信息（来自智能体1）
2. 依赖关系（来自智能体2）
3. Few-shot 示例（项目内历史用例）
4. RAG 上下文（相似接口的历史用例）

**输出代码**：
```python
import pytest
import requests

@pytest.fixture(scope="session")
def auth_token():
    # 登录逻辑
    response = requests.post(
        "https://api.example.com/api/login",
        json={"username": "test", "password": "test123"}
    )
    return response.json()["token"]

def test_create_order(auth_token):
    # 使用 token
    response = requests.post(
        "https://api.example.com/api/order",
        headers={"Authorization": f"Bearer {auth_token}"},
        json={"product_id": "123", "quantity": 1}
    )
    assert response.status_code == 201
    assert "order_id" in response.json()
```

**为什么需要 Few-shot + RAG？**
- Few-shot：模型先看到"好作文"，学会怎么写
- RAG：找到相似接口的历史用例，确保新用例风格一致

#### 流程5：智能体4 数据准备 → 排序 + 数据方案

**职责**：
在"真正跑测试前"，想好：
1. 用例要按什么顺序执行？（拓扑排序）
2. 每个用例的测试数据长什么样？

**拓扑排序 = 把依赖图转成执行顺序**

例子：
```
登录 → 创建订单 → 查询订单
```

这个顺序保证了：
- 登录必须先执行（获得token）
- 创建订单必须在登录之后（使用token）
- 查询订单必须在创建之后（使用order_id）

#### 流程6：智能体5 执行 & 报告 → Allure 报告

**执行逻辑**：

```
for 每个排序后的用例:
    1. 从前面用例的响应中提取数据
       (如：token、order_id)
    
    2. 填充到当前用例的请求中
       (headers 中放 token，path 中放 order_id)
    
    3. 执行请求
    
    4. 验证响应（断言）
    
    5. 提取响应数据，供下一个用例使用
```

**完整例子**：
```
step 1: 登录
  请求：{"username": "test", "password": "test123"}
  响应：{"token": "abc123"}
  提取：extracted_data["token"] = "abc123"

step 2: 创建订单（使用token）
  请求：{
    "headers": {"Authorization": "Bearer abc123"},
    "body": {"product_id": "123"}
  }
  响应：{"order_id": "456"}
  提取：extracted_data["order_id"] = "456"

step 3: 查询订单（使用order_id）
  请求：{
    "headers": {"Authorization": "Bearer abc123"},
    "path": "/api/order/456"
  }
  响应：{"status": "created"}
```

## 二、图谱化存储：为什么用 Neo4j？

### 2.1 为什么不用 MySQL？

**问题场景**：
```
你有3个接口：
- 接口A（登录）
- 接口B（创建订单）← 依赖A
- 接口C（查询订单）← 依赖B

问题：查询"接口C的完整依赖链"

用MySQL查询（递归困难）：
WITH RECURSIVE deps AS (
    SELECT target_interface_id FROM api_dependencies WHERE source_interface_id = 3
    UNION ALL
    SELECT d.target_interface_id FROM api_dependencies d
    JOIN deps ON d.source_interface_id = deps.target_interface_id
)
SELECT * FROM deps;

用Neo4j查询（一行代码）：
MATCH (target:APIInterface {id: '3'})<-[:DEPENDS_ON*]-(source:APIInterface)
RETURN source
```

**为什么Neo4j更优？**
1. ✅ 查询简单：Cypher语言天生支持图查询
2. ✅ 性能好：Neo4j专门优化了图遍历
3. ✅ 可视化：自带可视化工具，直观展示依赖关系
4. ✅ 支持复杂查询：多跳、路径查询、社区检测等

### 2.2 Neo4j 的核心概念

#### 节点（Node）
代表接口，包含属性：
```python
接口A:APIInterface {
  id: "login_001",
  name: "用户登录",
  method: "POST",
  path: "/api/login",
  crud_type: "LOGIN"
}
```

#### 关系（Relationship）
代表依赖关系，指向被依赖者：
```
创建订单 -[DEPENDS_ON]-> 登录
```

方向很重要：
- 箭头 → 代表"依赖方向"
- 创建订单 **依赖于** 登录

### 2.3 在 apitest 中如何存储？

**代码位置**：

#### 完整的Python代码示例：连接与存储

```python
from neo4j import GraphDatabase

class Neo4jStore:
    """Neo4j 存储和查询示例"""
    
    def __init__(self, uri, username, password):
        self.driver = GraphDatabase.driver(uri, auth=(username, password))
    
    def store_interfaces_and_dependencies(self, interfaces, dependencies, project_id):
        """
        将接口和依赖关系存储到Neo4j
        """
        with self.driver.session() as session:
            # 步骤1：创建接口节点
            for interface in interfaces:
                session.run("""
                    MERGE (api:APIInterface {
                        id: $id, 
                        project_id: $project_id
                    })
                    SET api.name = $name,
                        api.method = $method,
                        api.url = $url,
                        api.description = $description
                """, {
                    "id": interface["id"],
                    "project_id": project_id,
                    "name": interface["name"],
                    "method": interface["method"],
                    "url": interface["url"],
                    "description": interface.get("description", "")
                })
            
            # 步骤2：创建依赖关系
            for dep in dependencies:
                session.run("""
                    MATCH (source:APIInterface {id: $source_id})
                    MATCH (target:APIInterface {id: $target_id})
                    MERGE (source)-[r:DEPENDS_ON]->(target)
                    SET r.type = $type,
                        r.fields = $fields,
                        r.confidence = $confidence
                """, {
                    "source_id": dep["source_id"],
                    "target_id": dep["target_id"],
                    "type": dep.get("type", "parameter"),
                    "fields": dep.get("fields", []),
                    "confidence": dep.get("confidence", 0.9)
                })
    
    def get_all_dependencies(self, project_id):
        """
        查询项目的所有依赖关系
        
        返回: 所有的源→目标依赖对
        """
        with self.driver.session() as session:
            result = session.run("""
                MATCH (source:APIInterface)-[r:DEPENDS_ON]->(target:APIInterface)
                WHERE source.project_id = $project_id
                RETURN {
                    source_id: source.id,
                    source_name: source.name,
                    target_id: target.id,
                    target_name: target.name,
                    type: r.type,
                    fields: r.fields
                } as dependency
            """, {"project_id": project_id})
            
            return [record["dependency"] for record in result]
    
    def get_dependency_chain(self, target_api_id, project_id):
        """
        查询某个接口的完整依赖链（多跳）
        
        例: 查询"查询订单"接口的所有前置接口
        """
        with self.driver.session() as session:
            result = session.run("""
                MATCH path = (target:APIInterface {id: $target_id})
                          <-[:DEPENDS_ON*]-(ancestor:APIInterface)
                WHERE target.project_id = $project_id
                RETURN {
                    api_id: ancestor.id,
                    api_name: ancestor.name,
                    api_method: ancestor.method,
                    api_url: ancestor.url
                } as ancestor
                ORDER BY length(path)
            """, {
                "target_id": target_api_id,
                "project_id": project_id
            })
            
            return [record["ancestor"] for record in result]
    
    def close(self):
        self.driver.close()


# ===== 使用示例 =====
if __name__ == "__main__":
    # 初始化

        uri="bolt://localhost:7687",

    )

        {
            "id": "login_001",
            "name": "用户登录",

            "description": "使用用户名密码登录"
        },
        {
            "id": "order_001",

            "description": "创建新订单，需要token和userId"

            "id": "order_002",
            "name": "查询订单",
            "method": "GET",
            "url": "/api/order/{orderId}",
            "description": "查询订单详情"
        }
    ]
    
    # 要存储的依赖关系
    dependencies = [
        {
            "source_id": "order_001",
            "target_id": "login_001",
            "type": "parameter",
            "fields": ["token", "userId"],
            "confidence": 0.95
        },
        {
            "source_id": "order_002",
            "target_id": "order_001",
            "type": "parameter",
            "fields": ["orderId"],
            "confidence": 0.98
        }
    ]
    
    # 存储到Neo4j
    print("存储接口和依赖关系到Neo4j...")
    store.store_interfaces_and_dependencies(interfaces, dependencies, project_id=123)
    
    # 查询所有依赖
    print("\n查询所有依赖关系:")

        print(f"  {dep['source_name']} → {dep['target_name']} (字段: {dep['fields']})")
    
    # 查询某个接口的完整依赖链

    chain = store.get_dependency_chain("order_002", project_id=123)
    for ancestor in chain:
        print(f"  - {ancestor['api_name']} ({ancestor['api_method']} {ancestor['api_url']})")
    
    store.close()
    print("\n✅ 完成！")
```

## 三、拓扑排序：把"图"变成"执行顺序"

### 3.1 什么是拓扑排序？

**生活例子**：
```
做饭的步骤：
1. 买菜（无依赖）
2. 洗菜（依赖买菜）
3. 切菜（依赖洗菜）
4. 炒菜（依赖切菜和打鸡蛋）
5. 装盘（依赖炒菜）

问题：怎样安排这些步骤的执行顺序？
答案：拓扑排序
```

**在apitest中**：
```
三个测试用例：
1. 登录（无依赖）
2. 创建订单（依赖登录）
3. 查询订单（依赖创建订单）

拓扑排序结果：[登录, 创建订单, 查询订单]
```

### 3.2 拓扑排序算法：Kahn算法

**核心思想**：
```
1. 找到所有"没有依赖"的节点（入度为0）
2. 执行这些节点
3. 移除这些节点，更新其他节点的依赖
4. 重复1-3，直到所有节点都处理完
```

**入度的含义**：
```
入度0 = "没人依赖我，我可以先执行"
入度1 = "有1个人依赖我，我要等他们先执行我"
```

#### 完整的Python实现代码

```python
from collections import defaultdict, deque
from typing import List, Dict, Tuple

class TopologicalSorter:
    """拓扑排序实现 - Kahn算法"""
    
    def __init__(self):
        self.graph = defaultdict(list)  # 接口ID → 它依赖的接口列表
        self.in_degree = defaultdict(int)  # 接口ID → 入度
    
    def add_edge(self, source_id, target_id):
        """
        添加一条依赖边
        source 依赖于 target，即 source -> target
        """
        self.graph[target_id].append(source_id)  # target被source依赖
        self.in_degree[source_id] += 1  # source的入度+1
    
    def sort(self, all_ids: List[str]) -> Tuple[List[str], bool]:
        """
        进行拓扑排序
        
        返回: (排序结果, 是否有循环依赖)
        """
        # 初始化入度表
        in_degree = defaultdict(int)
        for api_id in all_ids:
            if api_id not in in_degree:
                in_degree[api_id] = 0
        
        # 计算每个节点的入度
        for api_id in all_ids:
            for dependent in self.graph[api_id]:
                in_degree[dependent] += 1
        
        # 初始化队列：入度为0的节点
        queue = deque([api_id for api_id in all_ids if in_degree[api_id] == 0])
        
        sorted_result = []
        
        # Kahn算法的核心循环
        while queue:
            # 1️⃣ 取出一个入度为0的节点
            current = queue.popleft()
            sorted_result.append(current)
            
            # 2️⃣ 移除该节点的出边，更新依赖于它的节点的入度
            for dependent in self.graph[current]:
                in_degree[dependent] -= 1
                
                # 3️⃣ 如果某个节点的入度变成0，加入队列
                if in_degree[dependent] == 0:
                    queue.append(dependent)
        
        # 检查是否有循环依赖
        has_cycle = len(sorted_result) != len(all_ids)
        
        return sorted_result, has_cycle


# ===== 使用示例 =====
if __name__ == "__main__":
    # 创建一个排序器
    sorter = TopologicalSorter()
    
    # 场景：测试电商平台的API
    apis = ["login", "getProduct", "addCart", "viewCart", "createOrder", "payOrder", "queryOrder"]
    
    # 添加依赖关系：source 依赖于 target
    # 说明：source必须在target之后执行
    dependencies = [
        ("getProduct", "login"),      # 获取商品需要登录
        ("addCart", "login"),         # 添加购物车需要登录
        ("addCart", "getProduct"),    # 添加购物车需要先查看商品
        ("viewCart", "addCart"),      # 查看购物车需要先添加
        ("createOrder", "viewCart"),  # 创建订单需要先查看购物车
        ("payOrder", "createOrder"),  # 支付需要先创建订单
        ("queryOrder", "payOrder")    # 查询订单需要先支付

    print("\n执行拓扑排序...")
    sorted_apis, has_cycle = sorter.sort(apis)

    else:
        print("✅ 排序成功！")
        print("\n执行顺序:")
        for i, api in enumerate(sorted_apis, 1):
            print(f"  {i}. {api}")
    
    print("\n详细执行流程:")
    print("  1️⃣ 登录（入度0，可以先执行）")
    print("  2️⃣ 获取商品（登录完后可执行，入度=1）")

    print("  4️⃣ 查看购物车（添加完后可执行，入度=1）")
```

### 3.3 算法演示

**代码位置**：
```python
backend/app/services/task_preparation.py
  └─ _sort_test_cases_by_dependency()
```

**核心代码**：

```python
def _sort_test_cases_by_dependency(self, test_cases, dependency_graph):
    """拓扑排序：根据依赖关系排序测试用例"""
    
    # 1. 构建依赖关系（用例级别）
    dependencies = {}  # case_id: [依赖的用例列表]
    for case in test_cases:
        api_id = case.api_interface_id
        dependencies[case.id] = [
            other.id for other in test_cases
            if is_dependent(api_id, other.api_interface_id)
        ]
    
    # 2. 计算入度（每个用例被多少个用例依赖）
    in_degree = {case.id: len(dependencies[case.id]) for case in test_cases}
    
    # 3. 初始化队列（找入度为0的用例）
    queue = [case.id for case.id, degree in in_degree.items() if degree == 0]
    
    # 4. Kahn算法的核心循环
    sorted_ids = []
    while queue:
        case_id = queue.pop(0)  # 取出一个入度为0的用例
        sorted_ids.append(case_id)  # 添加到结果
        
        # 更新依赖此用例的其他用例
        for dependent_id in get_dependents(case_id):
            in_degree[dependent_id] -= 1
            if in_degree[dependent_id] == 0:
                queue.append(dependent_id)  # 入度变0，可以执行了
    
    return sorted_ids
```

## 七、总结：为什么这样设计？

### 设计理念

1. **分离职责**：
   - 5个智能体各司其职
   - 每个智能体只做一件事，做到极致

2. **规则 + LLM 结合**：
   - 规则处理80%的情况（快+稳）
   - LLM处理20%的"聪明场景"（准+全）

3. **数据流驱动**：
   - Neo4j 存依赖关系（图结构天然适合）
   - 拓扑排序 确定执行顺序
   - 数据自动流转（token、id等）

4. **可插拔的RAG**：
   - Few-shot：让模型学会"怎么写"
   - RAG：让模型用上"之前怎么写的"
   - 比纯LLM更稳定、成本更低

### 对比其他方案

| 方案 | 优点 | 缺点 | apitest的选择 |
|------|------|------|---|
| 100%规则 | 快、稳定 | 不灵活 | ✗ |
| 100%LLM | 灵活 | 贵、慢、不稳定 | ✗ |
| 规则+LLM | 快、灵活、稳定 | 需要调优 | ✓ |
| 规则+RAG+LLM | 更好的生成质量 | 更复杂 | ✓ |

## 八、常见问题

**Q: 为什么依赖分析要分两步（规则+LLM）？**

A: 
- 规则 handles 80% 的情况，快速且可靠
- LLM 补充 20% 的复杂场景，灵活且准确
- 这样既保证速度，又保证质量

**Q: 为什么用Neo4j而不是Redis或MySQL？**

A:
- Redis：没有持久化和关系查询能力
- MySQL：递归查询复杂，性能差
- Neo4j：图数据库天生支持关系查询，性能好

**Q: 拓扑排序可能失败吗？**

A: 
- 如果存在循环依赖（A→B→C→A），会失败
- apitest的设计会避免产生循环依赖
- 如果真的产生了，会把剩余节点添加到末尾

**Q: 数据怎样在用例间流转？**

A:
```
1. 登录执行 → 响应中有 token
2. extracted_data = {"token": "abc123"}
3. 创建订单执行时：
   - 从 extracted_data 中取 token
   - 填充到请求头中
4. 创建订单响应 → 有 order_id
5. extracted_data = {..., "order_id": "456"}
6. 查询订单执行时：
   - 从 extracted_data 中取 order_id
   - 填充到URL路径中
```
