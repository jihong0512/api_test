## 智能体1：文档解析智能体

### 职责与目标
将用户上传的**各种格式的API文档**转换为**统一的结构化接口清单**。

### 输入输出

**输入**
```
- 用户上传的文档文件
  ├─ 格式：PDF、Word、Excel、JSON、YAML、TXT、Markdown、OpenAPI/Swagger
  ├─ 大小：通常 1KB ~ 10MB
  └─ 语言：支持中英文混合
```

**输出**
```
- 结构化接口清单
  ├─ 接口基本信息：名称、方法、URL、版本
  ├─ 请求定义：请求头、查询参数、请求体
  ├─ 响应定义：响应头、响应体、状态码
  └─ 元数据：标签、描述、业务含义
```

### 处理流程

```
上传文件
  ↓
[步骤1] 文件类型识别
  └─ 根据文件扩展名和内容特征识别格式
  
  ↓
[步骤2] 原始内容提取
  ├─ PDF → pdfplumber/PyPDF2 → 文本/表格
  ├─ Word → python-docx → 段落/表格
  ├─ JSON/YAML → json/yaml库 → 对象
  ├─ OpenAPI/Swagger → 直接解析标准格式
  └─ TXT/MD → 正则表达式 → 接口信息

  ↓
[步骤3] 内容理解与AI标准化
  ├─ 将提取的内容发送给LLM（DeepSeek/通义千问）
  ├─ LLM理解文档语义（即使格式不标准）
  └─ LLM统一输出为标准JSON格式

  ↓
[步骤4] 接口信息验证
  ├─ 检查必要字段完整性
  ├─ 验证HTTP方法有效性
  ├─ 验证URL格式正确性
  └─ 去重（相同接口只保留一条）

  ↓
[步骤5] 向量化存储
  ├─ 将每个接口的描述转换为向量（embedding）
  ├─ 存储到向量数据库（Milvus/ChromaDB）
  └─ 支持后续的语义搜索

  ↓
解析完成：返回接口清单
```

### 核心技术点

#### 多格式解析

| 格式 | 工具库 | 解析难点 | 解决方案 |
|------|--------|--------|--------|
| **PDF** | pdfplumber/PyPDF2 | 布局复杂，表格识别 | OCR + 结构分析 |
| **Word** | python-docx | 表格内容提取 | 递归遍历表格对象 |
| **Excel** | openpyxl/pandas | 多Sheet、合并单元格 | 遍历所有Sheet和行 |
| **JSON** | json库 | 嵌套结构解析 | 递归解析，支持Schema验证 |
| **YAML** | PyYAML | 锚点引用、流风格 | 完整解析包括引用展开 |
| **OpenAPI** | openapi-spec-validator | 版本差异（2.0/3.0） | 识别版本后分别处理 |
| **Markdown** | 正则表达式 | 不同的格式约定 | 支持多种表格/列表格式 |

#### LLM标准化的好处

```
原始文档内容（非常不规范）：
"用户登录功能，使用post方式调用https://api.example.com/login
需要传入用户名和密码，返回token
错误时返回error message"

           ↓ LLM理解和提取

标准化输出：

  "method": "POST",
  "url": "/login",
  "request": {
    "headers": {"Content-Type": "application/json"},
    "body": {

      "password": {"type": "string", "required": true}

  },
  "response": {
    "success": {
      "statusCode": 200,

    "error": {
      "statusCode": 400,
      "body": {"error": {"type": "string"}}

智能体1最终提取的接口信息包括：

**容错能力**

- 多语言 → 内部统一为英文，向用户显示原文

```

## 智能体2：依赖分析智能体

### 职责与目标
分析接口之间的**依赖关系**，构建**接口调用链路图**，规划**接口执行顺序**。

### 输入输出

**输入**
```
- 所有已解析的接口列表
  ├─ 接口基本信息（名称、方法、URL等）
  ├─ 请求参数定义
  └─ 响应数据定义
```

**输出**
```
- 依赖关系图 + 执行顺序
  ├─ 接口间的数据依赖关系
  ├─ 调用链路（A→B→C）
  ├─ 循环依赖检测与警告
  └─ 推荐的执行顺序（拓扑序列）
```

### 依赖分析方法

#### 方法1：规则分析（显式依赖）

```
CRUD顺序规则：
  Create (POST) → Read (GET) → Update (PUT) → Delete (DELETE)

同步操作规则：
  登录 (login) → 获取权限 (permission) → 操作资源 (resource)

场景识例：
  /user/register → /user/login → /user/profile → /user/logout
  参数关系：
    register响应中的userId
    ↓
    login需要的credentials（在register时创建）
    ↓
    profile可以使用login响应的token
```

#### 方法2：LLM语义分析（隐含依赖）

```
接口列表：
  1. "创建订单" (POST /order)
  2. "获取商品详情" (GET /product/:id)
  3. "查询库存" (GET /inventory)
  4. "支付订单" (POST /order/pay)

LLM分析：
  "创建订单需要知道商品信息和库存，所以创建订单前需要调用获取商品和查询库存"
  
  依赖关系：
    查询库存 ──┐
              ├─→ 创建订单 ──→ 支付订单
    获取商品详情 ──┘
```

#### 方法3：混合分析

```
结合规则和LLM：
  1. 规则识别显式的CRUD依赖
  2. LLM补充隐含的业务依赖
  3. 合并结果，构建完整的依赖图
```

### 构建依赖图

**依赖图数据结构**
```
图的节点 = 接口
图的边 = 依赖关系

例子：
    ┌─→ 获取商品详情
    │      ↑
    │      │ (前置)
    │      │
  用户 → 创建订单 → 支付订单 → 获取订单状态
    ↓         ↓          ↓
  (登录)  (需要商品信息) (需要订单ID)
```

### 拓扑排序与执行顺序

**拓扑排序的作用**
```
将接口按照依赖关系排序，使得每个接口都在其依赖的接口之后

排序前：A → B → C（依赖关系）
        D → C
        
排序后的执行顺序：D, A, B, C 或 A, D, B, C 等
保证：B执行时C已经完成，C执行时D已经完成
```

**Kahn算法实现拓扑排序**
```
算法步骤：
  1. 计算每个节点的入度（有多少个接口依赖它）
  2. 将入度为0的节点加入队列（没有依赖的接口可以先执行）
  3. 从队列中取出节点，加入排序结果
  4. 对于被取出节点指向的所有节点，入度减1
  5. 如果入度变为0，加入队列
  6. 重复直到队列为空

检测循环依赖：
  如果最终排序结果中的节点数 < 总节点数，说明有循环依赖
```

### Neo4j存储

为什么用Neo4j？
```
SQL数据库缺点：
  - 多级关系JOIN性能差
  - 复杂的递归查询困难
  
Neo4j优点：
  - 原生支持关系存储
  - 关系查询高效（毫秒级）
  - 可视化能力强
  - Cypher查询语言简洁易用
```

**Neo4j中的数据模型**
```
节点（Node）：
  (Interface {
    name: "用户登录",
    method: "POST",
    url: "/user/login",
    ...
  })

关系（Relationship）：
  (创建订单) -[:DEPENDS_ON]-> (查询库存)
  (支付订单) -[:DEPENDS_ON]-> (创建订单)
  (用户登录) -[:REQUIRES_TOKEN]-> (获取权限)
```

**Cypher查询示例**
```cypher
# 查找某个接口的所有前置接口（直接依赖）
MATCH (api:Interface {name: "支付订单"}) <- [:DEPENDS_ON] - (dep:Interface)
RETURN dep.name

# 查找某个接口的所有间接依赖
MATCH (api:Interface {name: "支付订单"}) <- [:DEPENDS_ON*] - (dep:Interface)
RETURN dep.name

# 查找整个调用链路
MATCH path = (api1:Interface) -[:DEPENDS_ON*]-> (api2:Interface)
WHERE api1.name = "注册用户" AND api2.name = "支付订单"
RETURN path
```

## 智能体3：测试用例生成智能体

### 职责与目标
基于接口信息、依赖关系和上下文，使用LLM**自动生成高质量的可执行测试用例**。

### 输入输出

**输入**
```
- 接口信息
  ├─ 接口定义（方法、URL、参数、响应）
  └─ 接口描述和业务含义

- 依赖关系
  ├─ 前置接口列表
  └─ 数据流转关系

- 上下文（Few-shot + RAG）
  ├─ 相似接口的测试用例示例
  └─ 项目历史的测试用例
```

**输出**
```
- 可执行的测试代码
  ├─ Pytest格式的Python代码
  ├─ 包含测试数据、请求、断言
  └─ 处理前置条件和数据流转
```

### 测试用例生成流程

```
输入：接口信息 + 依赖关系
  ↓
[步骤1] 分析接口参数
  ├─ 识别参数类型（string, number, boolean, object等）
  ├─ 识别是否必需
  └─ 识别参数的业务含义

  ↓
[步骤2] 构造测试数据
  ├─ 必需字段：构造有效值
  ├─ 可选字段：决定是否包含
  ├─ 边界值：计算max/min等
  └─ 特殊值：null, 空字符串, 超长值等

  ↓
[步骤3] 确定前置条件
  ├─ 根据依赖关系找出前置接口
  ├─ 确定需要从前置接口提取哪些数据
  └─ 规划数据流转：A的响应 → B的请求

  ↓
[步骤4] 调用LLM生成代码
  输入Prompt：
  """
  根据以下接口定义生成Pytest测试用例：
  接口：用户登录
  URL：POST /user/login
  请求参数：
    - username: string, required
    - password: string, required
  响应：
    - token: string
    - userId: int
  前置条件：需要先创建一个用户账号
  
  生成的测试用例应该：
  1. 清晰可读
  2. 包含有效的测试数据
  3. 设置适当的断言
  4. 处理错误情况
  """
  
  LLM输出：Python代码片段

  ↓
[步骤5] 代码验证和优化
  ├─ 语法检查（确保能运行）
  ├─ 逻辑检查（断言是否合理）
  ├─ 性能优化（简化代码）
  └─ 可读性优化（添加注释）

  ↓
生成完成：返回可执行的Pytest代码
```

### Few-shot学习

**什么是Few-shot**
```
从几个示例中学习模式，然后生成新内容

例子：
  想让LLM为新接口生成测试用例
  ├─ 方法1（0-shot）：直接给接口定义，要求生成 → 质量一般
  ├─ 方法2（Few-shot）：先给2-3个高质量示例，然后要求为新接口生成 → 质量更好
  └─ Few-shot：Prompt里包含 "参考以下示例的格式和思路"
```

**Few-shot在项目中的应用**
```
Prompt模板：

你是一个API测试专家。根据以下示例生成测试用例：

## 示例1：用户登录接口的测试用例
[完整的高质量测试代码示例]

## 示例2：用户信息更新接口的测试用例
[另一个高质量测试代码示例]

## 现在为以下接口生成测试用例
接口：[新接口定义]

参考上面的示例格式，为该接口生成类似质量的测试用例：
```

### RAG检索增强

**什么是RAG**
```
RAG = Retrieval-Augmented Generation
检索增强生成：在生成内容前，先检索相关的上下文，作为参考
```

**在测试用例生成中的应用**
```
步骤：
  1. 用户要为"订单支付"接口生成测试用例
  2. 系统从向量数据库检索：
     - 相似的接口（如"订单创建"）的测试用例
     - 历史的支付相关测试用例
     - 支付接口的通用最佳实践
  3. 将检索结果作为Few-shot示例
  4. 调用LLM生成新的测试用例

好处：
  ✅ 新接口继承了旧接口的最佳实践
  ✅ 测试风格保持一致
  ✅ 自动学习项目特定的测试约定
```

### 测试用例内容

生成的Pytest测试用例包含：

```python
import pytest
import requests

class TestUserLogin:
    """用户登录接口测试"""
    
    # 必需的测试数据
    @pytest.fixture
    def login_data(self):
        return {
            "username": "test_user",
            "password": "password123"
        }
    
    # 正常场景测试
    def test_login_success(self, login_data):
        response = requests.post(
            url="https://api.example.com/user/login",
            json=login_data
        )
        assert response.status_code == 200
        assert "token" in response.json()
        assert "userId" in response.json()
    
    # 边界值测试
    def test_login_empty_username(self):
        response = requests.post(
            url="https://api.example.com/user/login",
            json={"username": "", "password": "pwd"}
        )
        assert response.status_code == 400
    
    # 错误场景测试
    def test_login_wrong_password(self, login_data):
        login_data["password"] = "wrongpassword"
        response = requests.post(...)
        assert response.status_code == 401
    
    # 依赖的前置条件
    @pytest.mark.dependency
    def test_user_register_first(self):
        """用户登录前需要先注册"""
        # 调用注册接口
        register_response = requests.post(...)
        assert register_response.status_code == 201
        
        # 提取userId用于后续测试
        pytest.user_id = register_response.json()["userId"]
```

## 智能体4：数据准备与执行智能体

### 职责与目标
为测试执行**做数据准备**，按照**正确的顺序**执行测试，**自动传递数据**流转。

### 输入输出

**输入**
```
- 测试用例代码
  └─ 所有生成的Pytest测试代码

- 依赖关系图
  ├─ 接口执行顺序
  └─ 数据流转关系

- 执行配置
  ├─ 测试环境（开发/测试/预发环布）
  ├─ 超时时间
  └─ 重试次略
```

**输出**
```
- 测试执行结果
  ├─ 每个接口的执行状态（通过/失败）
  ├─ 执行日志和错误信息
  └─ Allure格式的详细报告
```

### 数据流转机制

**场景：多个接口有依赖关系**

```
业务流程：
  1. 用户注册 → 返回userId
  2. 用户登录 → 需要username/password，返回token
  3. 创建订单 → 需要token和userId
  4. 支付订单 → 需要订单ID，返回支付结果

数据流转：
  注册响应的userId
        ↓ (自动提取和传递)
  用于登录请求和创建订单请求
        ↓
  创建订单响应的订单ID
        ↓ (自动提取和传递)
  用于支付订单请求
```

**占位符替换机制**

```
测试代码中的占位符：
  {user_id}  → 替换为注册响应中的userId
  {token}    → 替换为登录响应中的token
  {order_id} → 替换为创建订单响应中的orderId

执行前处理：
  原始：
    "user_id": "{user_id}",
    "Authorization": "Bearer {token}"
  
  替换后（从前置接口响应中提取）：
    "user_id": "12345",
    "Authorization": "Bearer eyJhbGciOiJIUzI1NiIs..."
```

### 执行流程

```
输入：所有测试用例 + 依赖图
  ↓
[步骤1] 拓扑排序
  └─ 按照依赖关系重新排序测试，保证前置接口先执行

  ↓
[步骤2] 环境初始化
  ├─ 初始化数据库（清空测试数据或重置到已知状态）
  ├─ 初始化HTTP客户端（设置超时、重试等）
  └─ 初始化日志系统

  ↓
[步骤3] 按序执行测试
  对每个测试：
    1. 从之前的测试结果中提取需要的数据
    2. 替换占位符
    3. 执行HTTP请求
    4. 收集响应
    5. 执行断言
    6. 保存响应供后续测试使用

  ↓
[步骤4] 错误处理
  如果某个测试失败：
    ├─ Option 1: 停止执行（某些关键接口失败时）
    ├─ Option 2: 继续执行（记录失败，继续）
    └─ Option 3: 重试（网络抖动时）

  ↓
[步骤5] 汇总结果
  ├─ 统计成功/失败/跳过的测试数
  ├─ 生成Allure报告
  └─ 保存日志

  ↓
返回：测试执行报告
```

### 关键技术点

**异步执行**
```
可以并行执行没有依赖关系的接口：

依赖关系：
  A → B
  A → C

执行方案（串行）：
  A (1秒) → B (1秒) → C (1秒) = 3秒

执行方案（并行）：
  A (1秒) → [B和C并行] (1秒) = 2秒

使用Celery/asyncio实现并行执行
```

**字段提取**
```
从复杂的嵌套JSON响应中提取字段：

响应：
{
  "code": 200,
  "data": {
    "user": {
      "id": 12345,
      "profile": {
        "name": "张三"
      }
    }
  }
}

提取规则：
  data.user.id → 12345
  data.user.profile.name → "张三"

使用JSONPath或点号路径访问嵌套字段
```

## 智能体5：错误分析与报告智能体

### 职责与目标
分析测试失败的**根本原因**，生成**智能化的分析报告**。

### 输入输出

**输入**
```
- 测试执行日志
  ├─ 每个接口的请求和响应
  ├─ 断言失败的信息
  └─ 异常堆栈

- 测试用例代码
  └─ 每个失败用例的源代码

- 系统环境信息
  ├─ API服务器状态
  ├─ 数据库状态
  └─ 网络连接状态
```

**输出**
```
- 智能分析报告
  ├─ 失败原因分析（LLM自动识别）
  ├─ 是否是API bug或测试用例有问题
  ├─ 解决建议
  └─ 优化方案

- 视觉化报告
  ├─ HTML格式的测试报告
  ├─ Allure报告
  └─ 仪表板和趋势图
```

### 错误分析方法

**常见错误类型和分析**

```
1️⃣ 业务逻辑错误（真正的Bug）
   特征：
     - 响应状态码正确，但返回数据不符合预期
     - 断言失败：expected X, got Y
   
   分析方法：
     - 比对API文档和实际响应
     - 检查数据库中的真实数据
     - 通过SQL查询验证数据

2️⃣ 接口未实现（开发进度问题）
   特征：
     - 返回404或501
     - "接口不存在"或"未实现"的错误信息
   
   分析方法：
     - 检查API服务的启动状态
     - 查看最近的代码提交
     - 联系后端开发确认

3️⃣ 数据依赖问题（测试数据问题）
   特征：
     - 外键约束失败
     - 找不到指定的资源（404）
     - "用户不存在"等业务错误
   
   分析方法：
     - 检查前置接口是否成功执行
     - 验证数据是否正确流转
     - 手动查询数据库验证

4️⃣ 网络或超时问题（环境问题）
   特征：
     - Connection timeout / Connection refused
     - 504 Gateway Timeout
   
   分析方法：
     - 检查API服务是否正常
     - 网络连接诊断
     - 增加超时时间重试

5️⃣ 测试用例问题（测试质量问题）
   特征：
     - 测试数据不符合业务规则
     - 断言条件过于严格
     - 时间相关的竞态条件
   
   分析方法：
     - 查看测试代码逻辑
     - 对比其他正常的用例
     - 调整测试数据或断言
```

### 生成分析报告

**LLM分析过程**

```
输入一个失败的测试：
  
  接口：支付订单
  请求：POST /order/pay with {"orderId": 12345}
  期望响应：{"code": 200, "data": {"paymentId": "xxx"}}
  实际响应：{"code": 200, "data": {"paymentId": null}}

LLM分析：
  "paymentId为null表示支付流程虽然返回成功，但没有生成支付ID。
   这通常说明：
   1. 后端支付逻辑有bug
   2. 或者订单数据不完整（缺少必要的商品信息）
   建议：
   1. 检查后端支付服务的日志
   2. 验证订单中的商品是否有效
   3. 确认订单状态是否允许支付"
```

### 报告生成

**Allure报告**
```
优点：
  - 美观的Web界面
  - 详细的执行历史和趋势
  - 支持截图和附件
  - 支持标签和分类

内容：
  - 总体统计（成功率、耗时等）
  - 用例详情（步骤、日志、失败原因）
  - 环境信息（API版本、数据库等）
  - 历史趋势（通过率变化）
```

## 5个智能体的完整协作

### 核心流程

```
┌─────────────────────────────────────────────────────────────────┐
│                        用户上传文档                               │
└─────────────────┬───────────────────────────────────────────────┘
                  │
                  ▼
        ┌─────────────────────┐
        │  智能体1：文档解析   │ ─→ 输出：接口清单
        └──────────┬──────────┘
                   │
                   ▼
        ┌─────────────────────┐
        │  智能体2：依赖分析   │ ─→ 输出：依赖图 + 执行顺序
        └──────────┬──────────┘
                   │
                   ▼
        ┌─────────────────────┐
        │  智能体3：用例生成   │ ─→ 输出：测试代码
        └──────────┬──────────┘
                   │
                   ▼
        ┌─────────────────────┐
        │  智能体4：数据流转   │ ─→ 输出：执行结果
        │     和执行           │
        └──────────┬──────────┘
                   │
                   ▼
        ┌─────────────────────┐
        │  智能体5：错误分析   │ ─→ 输出：分析报告
        │      和报告          │
        └─────────────────────┘
```

### 协作特点

1. **串行流程**：前一个智能体的输出是下一个智能体的输入
2. **数据流动**：信息在各智能体间传递，不断增加价值
3. **反馈循环**：如果报告发现bug，开发者修复后重新运行流程
4. **独立性**：每个智能体是独立的模块，可以单独测试或替换

### 使用场景

**场景1：初次上传文档**
```
步骤：
  1. 用户上传API文档（如Swagger JSON）
  2. 智能体1快速完成解析（几秒）
  3. 智能体2分析依赖关系（几秒）
  4. 智能体3生成完整测试套件（几分钟，因为调用LLM）
  5. 用户审查和修改生成的用例
  6. 用户点击"运行测试"
  7. 智能体4执行所有测试（几分钟）
  8. 智能体5生成报告（几秒）
  
总耗时：5-10分钟
```

**场景2：接口文档更新**
```
步骤：
  1. 开发者更新了API文档
  2. 用户重新上传更新的文档
  3. 智能体1解析新文档（检测到新增接口）
  4. 智能体2更新依赖关系
  5. 智能体3为新增接口生成用例
  6. 用户运行所有测试（包括新增用例）
  7. 自动检测新增接口的问题
  
目的：持续验证API的完整性
```

**场景3：定期回归测试**
```
步骤：
  1. 系统每天定时运行所有测试（使用已有的测试用例）
  2. 智能体4执行所有用例
  3. 智能体5分析结果，发现哪些接口变得不稳定
  4. 生成日报或周报
  
目的：及时发现API性能或功能的回归
```

## 总结

| 智能体 | 主要职责 | 关键技术 | 输入 | 输出 |
|--------|--------|--------|------|------|
| **智能体1** | 文档解析 | 多格式解析库 + LLM | 各种格式的文档 | 接口清单 |
| **智能体2** | 依赖分析 | 规则识别 + LLM语义 + Neo4j + Kahn算法 | 接口清单 | 依赖图 + 执行顺序 |
| **智能体3** | 用例生成 | Few-shot + RAG + LLM | 接口 + 依赖 + 上下文 | 测试代码 |
| **智能体4** | 数据执行 | 拓扑排序 + 占位符替换 + 异步执行 | 测试代码 + 依赖图 | 执行结果 |
| **智能体5** | 错误分析 | LLM分析 + Allure报告生成 | 执行日志 + 代码 | 分析报告 |

这5个智能体协作完成了**从文档到测试报告**的完整自动化流程！
