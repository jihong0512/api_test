## 依赖分析概述

### 核心问题

```
问题场景：
  系统有100个接口，它们之间有很多依赖关系
  
  如果直接运行所有接口的测试，会出现：
    ❌ 某个接口失败（缺少前置数据）
    ❌ 某个接口超时（等待数据）
    ❌ 测试报告混乱（无法判断真正的问题）
  
  解决方案：
    ✅ 分析接口间的依赖关系
    ✅ 规划正确的执行顺序
    ✅ 自动化数据流转
```

### 依赖分析的作用

| 作用 | 说明 | 场景 |
|------|------|------|
| **执行顺序规划** | 确定接口测试的执行顺序 | 避免数据不存在导致失败 |
| **数据流转** | 自动从前置接口提取数据 | 减少手工构造测试数据 |
| **业务流程识别** | 识别完整的业务流程 | 进行端到端测试 |
| **影响分析** | 发现一个接口改动的影响范围 | 确定回归测试范围 |

## 4种依赖类型

### 1️⃣ 参数依赖（数据流依赖）

**定义**：一个接口的**输入参数**来自另一个接口的**响应数据**

**识别方式**：
```
Source API 的响应字段
        ↓ (数据流)
Target API 的请求参数
```

#### 示例：登录后获取用户信息

**接口A - 登录**：
```json
POST /api/login
请求: {username: "user1", password: "pwd123"}
响应: {
  "code": 0,
  "data": {
    "token": "eyJ9...",        ← 这个字段是关键！
    "userId": 123,              ← 这个字段是关键！
    "username": "user1"
  }
}
```

**接口B - 获取用户信息**：
```json
GET /api/users/{userId}
请求头: {Authorization: "Bearer {token}"}    ← 需要来自A的token
路径参数: {userId: ???}                       ← 需要来自A的userId
响应: {
  "code": 0,
  "data": {
    "id": 123,
    "username": "user1",
    "email": "user@example.com"
  }
}
```

**依赖关系**：
```
接口B 依赖于 接口A

数据映射：
  A.response.data.token    → B.headers.Authorization
  A.response.data.userId   → B.pathParam.userId
```

#### 常见的参数对应模式

| 源字段 | 目标参数位置 | 说明 |
|--------|-----------|------|
| `token` / `accessToken` | `Authorization` 请求头 | 认证令牌 |
| `userId` / `uid` | 路径参数 `{userId}` 或查询参数 | 用户ID |
| `id` / `objectId` | 路径参数 `{id}` | 资源ID |
| `orderId` | 路径参数或请求参数 | 订单ID |
| `sessionId` | Cookie 或请求头 | 会话ID |

#### Python识别代码

```python
def find_parameter_dependencies(interfaces):
    """
    识别接口间的参数依赖
    """
    dependencies = []
    
    for target_api in interfaces:
        # 收集target_api需要的所有参数
        target_params = set()
        target_params.update(extract_header_params(target_api))
        target_params.update(extract_path_params(target_api))
        target_params.update(extract_query_params(target_api))
        target_params.update(extract_body_params(target_api))
        
        # 逐个检查其他接口的响应
        for source_api in interfaces:
            if source_api == target_api:
                continue
            
            # 收集source_api响应中的所有字段
            source_fields = extract_response_fields(source_api)
            
            # 找出匹配的字段
            matching = target_params & source_fields
            if matching:
                dependencies.append({
                    'source': source_api['name'],
                    'target': target_api['name'],
                    'type': 'parameter',
                    'mapped_fields': list(matching),
                    'confidence': 0.95 if len(matching) >= 2 else 0.7
                })
    
    return dependencies

def extract_response_fields(api):
    """提取API响应中的所有字段名"""
    fields = set()
    
    response = api.get('response', {})
    body = response.get('body', {})
    
    def collect_fields(obj, prefix=''):
        if isinstance(obj, dict):
            for key, value in obj.items():
                full_key = f"{prefix}.{key}" if prefix else key
                fields.add(full_key)
                fields.add(key)  # 也加入简单名
                if isinstance(value, dict):
                    collect_fields(value, full_key)
        elif isinstance(obj, list) and obj:
            collect_fields(obj[0], prefix)
    
    collect_fields(body)
    return fields
```

### 2️⃣ 业务流程依赖（顺序依赖）

**定义**：多个接口组成一个完整的业务流程，必须按特定顺序执行

**识别方式**：
```
根据业务逻辑，识别自然的执行流程
```

#### 示例：电商订单流程

**流程描述**：
```
1. 用户登录
   POST /auth/login
   
2. 浏览商品列表
   GET /products
   
3. 查看商品详情
   GET /products/{productId}
   
4. 加入购物车
   POST /cart/items
   
5. 查看购物车
   GET /cart
   
6. 创建订单
   POST /orders
   
7. 支付订单
   POST /orders/{orderId}/pay
   
8. 查询订单状态
   GET /orders/{orderId}
```

**业务流程依赖图**：
```
登录
  ↓
  └─→ 浏览商品 ──→ 查看详情
       ↓
       ├─→ 加入购物车 ──→ 查看购物车 ──→ 创建订单 ──→ 支付 ──→ 查询
```

**特点**：
- 不是每个相邻接口都有参数依赖，但它们有逻辑依赖
- 必须按顺序执行以模拟真实的用户操作
- 跳过某些步骤（如"浏览商品"）可能也能工作，但不符合真实场景

#### LLM识别代码

```python
async def find_business_flow_dependencies(interfaces, llm_service):
    """
    使用LLM识别业务流程依赖
    """
    
    interfaces_text = json.dumps(
        [{
            'name': api['name'],
            'method': api['method'],
            'url': api['url'],
            'description': api.get('description', '')
        } for api in interfaces],
        ensure_ascii=False,
        indent=2
    )
    
    prompt = f"""
    分析以下接口的业务逻辑，识别完整的业务流程和流程依赖。
    
    接口列表：
    {interfaces_text}
    
    请识别：
    1. 完整的业务流程（如：登录 → 浏览商品 → 下单 → 支付）
    2. 各个流程中接口的依赖顺序
    3. 可选的支路（如：登录后可以直接下单，也可以先浏览商品）
    
    返回JSON格式的分析结果：
    {{
        "business_flows": [
            {{
                "name": "流程名称",
                "description": "流程说明",
                "sequence": ["接口1", "接口2", "接口3"],
                "critical": true  // 是否是关键业务流程
            }}
        ],
        "flow_dependencies": [
            {{
                "source": "接口A",
                "target": "接口B",
                "reason": "为什么B依赖于A的业务说明"
            }}
        ]
    }}
    """
    
    result = await llm_service.chat(prompt, temperature=0.1)
    return json.loads(result)
```

### 3️⃣ 资源依赖（前置创建）

**定义**：接口操作的资源需要先被创建才能使用

**识别方式**：
```
CRUD操作的依赖关系：
  Create (创建) 必须在 Read/Update/Delete 之前
```

#### 示例：用户管理API

**资源**：用户（User）

```
创建用户      POST /users
  ↑
  |（必须先创建）
  |
  ├─→ 读取用户    GET /users/{userId}
  ├─→ 更新用户    PUT /users/{userId}
  └─→ 删除用户    DELETE /users/{userId}
```

**CRUD依赖规则**：
```
Create  →  Read/Update/Delete (读/改/删都需要资源先存在)
Update  →  Delete (更新后才能删除)
Read    →  (无依赖，可独立执行)
```

#### 识别代码

```python
def find_crud_dependencies(interfaces):
    """
    识别CRUD操作间的资源依赖
    """
    
    # 按资源分组
    resources = {}
    
    for api in interfaces:
        # 从URL提取资源名
        # 例：/users → users, /products → products
        resource = extract_resource_from_url(api['url'])
        
        if resource not in resources:
            resources[resource] = {'C': None, 'R': [], 'U': [], 'D': None}
        
        # 按HTTP方法分类
        method = api['method'].upper()
        if method == 'POST':
            resources[resource]['C'] = api
        elif method == 'GET':
            resources[resource]['R'].append(api)
        elif method in ['PUT', 'PATCH']:
            resources[resource]['U'].append(api)
        elif method == 'DELETE':
            resources[resource]['D'] = api
    
    # 生成依赖关系
    dependencies = []
    for resource, operations in resources.items():
        create_api = operations['C']
        
        if create_api:
            # Create必须在R/U/D之前
            for read_api in operations['R']:
                dependencies.append({
                    'source': create_api['name'],
                    'target': read_api['name'],
                    'type': 'resource',
                    'reason': f"必须先创建{resource}才能读取"
                })
            
            for update_api in operations['U']:
                dependencies.append({
                    'source': create_api['name'],
                    'target': update_api['name'],
                    'type': 'resource',
                    'reason': f"必须先创建{resource}才能更新"
                })
            
            if operations['D']:
                dependencies.append({
                    'source': create_api['name'],
                    'target': operations['D']['name'],
                    'type': 'resource',
                    'reason': f"必须先创建{resource}才能删除"
                })
    
    return dependencies

def extract_resource_from_url(url):
    """从URL提取资源名"""
    # 例：/api/users → users, /api/users/{id} → users
    parts = url.split('/')
    for part in reversed(parts):
        if part and not part.startswith('{'):
            return part
    return 'resource'
```

### 4️⃣ 认证依赖

**定义**：接口需要有效的认证令牌才能访问

**识别方式**：
```
检查接口是否需要：
  - Authorization 请求头
  - token 参数
  - Cookie 中的 session
```

#### 示例：需要认证的接口

```python
def find_authentication_dependencies(interfaces):
    """
    识别需要认证的接口
    """
    
    # 首先找到登录接口（通常会返回token）
    login_api = None
    for api in interfaces:
        if any(keyword in api['name'].lower() for keyword in ['login', 'auth', '认证']):
            login_api = api
            break
    
    if not login_api:
        return []
    
    # 找出所有需要认证的接口
    dependencies = []
    
    for api in interfaces:
        if api == login_api:
            continue
        
        # 检查是否需要认证
        requires_auth = False
        
        # 方法1：检查是否有Authorization头
        headers = api.get('request', {}).get('headers', {})
        if 'Authorization' in headers or 'authorization' in headers:
            requires_auth = True
        
        # 方法2：检查是否有token参数
        params = api.get('request', {}).get('params', {})
        if any('token' in str(k).lower() for k in params.keys()):
            requires_auth = True
        
        # 方法3：根据URL模式判断（通常/api/开头的接口都需要认证）
        if '/api/' in api['url'] and api['method'] != 'GET':
            requires_auth = True
        
        if requires_auth:
            dependencies.append({
                'source': login_api['name'],
                'target': api['name'],
                'type': 'authentication',
                'reason': '需要有效的认证令牌'
            })
    
    return dependencies
```

## 3种分析方法

### 方法1：启发式规则分析（快速但可能不完整）

**原理**：
- 通过参数名匹配
- 通过HTTP方法类型（CRUD）
- 通过URL模式

**优点**：
- ✅ 快速（毫秒级）
- ✅ 无需额外的模型调用
- ✅ 结果可预测

**缺点**：
- ❌ 容易误判（参数名相同但无关）
- ❌ 无法识别复杂的业务逻辑
- ❌ 对参数命名敏感

**代码示例**：
```python
def rule_based_analysis(interfaces):
    """基于规则的依赖分析"""
    
    dependencies = []
    
    # 规则1：参数名称匹配
    dependencies.extend(find_parameter_dependencies(interfaces))
    
    # 规则2：CRUD顺序
    dependencies.extend(find_crud_dependencies(interfaces))
    
    # 规则3：认证需求
    dependencies.extend(find_authentication_dependencies(interfaces))
    
    return dependencies
```

### 方法2：LLM语义分析（准确但成本高）

**原理**：
- 使用大模型理解接口语义
- 理解业务逻辑关系
- 识别隐含的依赖

**优点**：
- ✅ 准确度高（85%+）
- ✅ 能识别复杂依赖
- ✅ 能识别业务流程
- ✅ 能进行合理推断

**缺点**：
- ❌ 调用成本高（API费用）
- ❌ 响应时间长（1-5秒）
- ❌ 可能出现幻觉（编造依赖）

**代码示例**：
```python
async def llm_based_analysis(interfaces, llm_service):
    """基于LLM的依赖分析"""
    
    interfaces_text = json.dumps(interfaces, ensure_ascii=False, indent=2)
    
    prompt = f"""
    分析以下API接口的依赖关系。请识别：
    
    1. 参数依赖（数据流）
    2. 业务流程依赖（顺序关系）
    3. 资源依赖（CRUD顺序）
    4. 认证依赖
    
    接口列表：
    {interfaces_text}
    
    请返回结构化的分析结果...
    """
    
    result_text = await llm_service.chat(prompt, temperature=0.1)
    result = json.loads(result_text)
    
    return result
```

### 方法3：混合分析（推荐 - 既快又准）

**原理**：
- 先用规则快速分析
- 再用LLM验证和补全
- 综合两种方法的优点

**流程**：
```
步骤1：启发式规则分析（快速）
  ↓
步骤2：LLM验证和补全（精准）
  ↓
步骤3：合并和去重
  ↓
最终的完整依赖关系
```

**代码示例**：
```python
async def hybrid_analysis(interfaces, llm_service):
    """混合分析：规则 + LLM"""
    
    # 步骤1：规则分析
    print("步骤1：执行规则分析...")
    rule_deps = rule_based_analysis(interfaces)
    print(f"  识别出 {len(rule_deps)} 个依赖")
    
    # 步骤2：准备LLM输入
    rule_deps_text = json.dumps(rule_deps, ensure_ascii=False, indent=2)
    interfaces_text = json.dumps(interfaces, ensure_ascii=False, indent=2)
    
    prompt = f"""
    以下是通过规则初步分析出的接口依赖关系：
    {rule_deps_text}
    
    原始接口列表：
    {interfaces_text}
    
    请验证这些依赖是否正确，并补充遗漏的依赖关系。
    特别注意：
    1. 业务流程中的隐含依赖
    2. 参数名不同但语义相关的字段映射
    3. 需要认证的接口
    
    返回完整的依赖分析结果...
    """
    
    # 步骤3：LLM验证和补全
    print("步骤2：执行LLM分析...")
    llm_result_text = await llm_service.chat(prompt, temperature=0.1)
    llm_deps = json.loads(llm_result_text)
    
    # 步骤4：合并去重
    print("步骤3：合并结果...")
    all_deps = merge_dependencies(rule_deps, llm_deps['dependencies'])
    
    return all_deps

def merge_dependencies(rule_deps, llm_deps):
    """合并并去重依赖关系"""
    
    # 使用元组作为key，便于去重
    seen = set()
    merged = []
    
    for dep in rule_deps + llm_deps:
        key = (dep['source'], dep['target'], dep['type'])
        if key not in seen:
            seen.add(key)
            merged.append(dep)
    
    return merged
```

**混合分析的优势**：
- ✅ 快速：规则分析只需毫秒
- ✅ 准确：LLM补充和验证
- ✅ 成本低：LLM只用于补全，而不是全量分析
- ✅ 可信：规则分析的结果有明确的逻辑支持

## 最佳实践

### ✅ 推荐做法

**1. 使用混合方法**
```python
# 推荐：规则 + LLM
deps = await hybrid_analysis(interfaces, llm_service)
```

**2. 明确标注依赖类型**
```python
{
    'source': '登录',
    'target': '获取用户信息',
    'type': 'parameter',  # 清晰的类型
    'reason': '需要token'
}
```

**3. 标注置信度**
```python
{
    'source': '创建订单',
    'target': '支付订单',
    'type': 'business_flow',
    'confidence': 0.9  # 90%置信度
}
```

**4. 定期验证和更新**
```python
# 定期运行分析，对比和验证
old_deps = load_previous_analysis()
new_deps = await hybrid_analysis(interfaces, llm_service)

# 检查差异
diff = compare_dependencies(old_deps, new_deps)
if diff['removed']:
    print(f"警告：{len(diff['removed'])} 个依赖被移除")
if diff['added']:
    print(f"新增：{len(diff['added'])} 个依赖")
```

### ❌ 避免的做法

**1. 只使用参数名匹配**
```python
# ❌ 不完整，容易误判
deps = find_parameter_dependencies(interfaces)
```

**2. 不标注置信度**
```python
# ❌ 无法判断哪些依赖可靠
dep = {
    'source': '接口A',
    'target': '接口B'
}
```

**3. 一次性分析，之后不更新**
```python
# ❌ 随着API变化，依赖关系也会变
# 应该定期重新分析
```

## 总结

| 方法 | 优点 | 缺点 | 应用场景 |
|------|------|------|---------|
| **规则分析** | 快速、可预测 | 容易误判、不完整 | 快速初步分析 |
| **LLM分析** | 准确、全面 | 成本高、响应慢 | 准确度要求高 |
| **混合分析** | 快速、准确 | 实现复杂 | ⭐ **推荐** |

**推荐工作流**：
```
1. 使用规则快速分析 → 获得初步依赖关系
2. 用LLM验证和补全 → 确保准确和完整
3. 标注类型和置信度 → 便于后续使用
4. 定期重新分析 → 保持与API同步
```
