# 小场景测试用例集（TestCaseSuite）生成逻辑说明

## 概述

小场景测试用例集（`TestCaseSuite`）是系统中用于组织和管理接口测试场景的数据结构。每个场景用例集包含一组按依赖关系排序的接口，用于生成场景化的测试用例。

## 生成时机

小场景测试用例集有两种生成方式：

### 方式1：自动生成（推荐）

**触发时机：** 运行"全局接口依赖分析"时自动生成

**流程：**
1. 用户在"接口依赖分析"页面点击"分析所有接口依赖"按钮
2. 调用 `POST /api/api-dependencies/analyze-all/{project_id}` 接口
3. 后端启动 `analyze_all_interfaces_task` Celery异步任务
4. 在依赖分析过程中，`OptimizedDependencyAnalyzer.analyze_interfaces()` 方法会：
   - 对接口进行相似度分组
   - 使用LLM分析接口间的依赖关系
   - 构建依赖链拓扑图
   - 生成场景用例集数据（`all_scenarios`）
5. 调用 `_store_scenarios_to_db_and_redis()` 方法（位于 `optimized_dependency_analyzer.py:2849`）：
   - 将场景用例集存储到数据库 `test_case_suites` 表
   - 将场景用例集数据存储到Redis（key: `project:{project_id}:scenarios`）
   - 存储到Neo4j图数据库（可选）

**关键代码位置：**
- `backend/app/routers/api_dependencies.py:403` - `analyze_all_interfaces` 接口
- `backend/app/celery_tasks.py:3344` - `analyze_all_interfaces_task` Celery任务
- `backend/app/services/optimized_dependency_analyzer.py:1844` - 调用存储方法
- `backend/app/services/optimized_dependency_analyzer.py:2849` - `_store_scenarios_to_db_and_redis` 实现

**存储的数据结构：**
- `test_case_suites` 表的 `test_case_ids` 字段存储JSON格式的接口ID列表
- 每个场景用例集的接口列表以 `['__LOGIN_INTERFACE__', interface_id1, interface_id2, ...]` 格式存储
- 第一个元素 `__LOGIN_INTERFACE__` 表示登录接口（特殊标识）
- 后续接口按CRUD顺序排列：创建 -> 修改 -> 查询 -> 删除

### 方式2：手动生成

**触发时机：** 用户在前端"小场景用例集"页面手动点击"从依赖分析结果生成小场景用例集"按钮

**流程：**
1. 用户在前端"小场景用例集"页面点击"从依赖分析结果生成小场景用例集"按钮
2. 调用 `POST /api/test-case-suites/generate-from-dependencies/{project_id}` 接口
3. 后端 `generate_scenario_suites_from_dependencies()` 函数（位于 `test_case_suites.py:644`）会：
   - 清除该项目的现有场景用例集数据（Redis、ChromaDB、Neo4j、数据库）
   - 从Redis或Neo4j读取接口依赖分析结果
   - 如果Redis和Neo4j都没有数据，则从数据库元数据重新分析
   - 基于依赖分析结果重新生成场景用例集
   - 存储到数据库、Redis、Neo4j、ChromaDB

**关键代码位置：**
- `backend/app/routers/test_case_suites.py:644` - `generate_scenario_suites_from_dependencies` 接口
- `frontend/src/pages/ScenarioSuites.js:76` - `handleGenerateFromDependencies` 函数

**使用场景：**
- 依赖分析已完成，但需要重新生成场景用例集
- 依赖分析结果发生变化，需要更新场景用例集
- 自动生成失败后，手动重新生成

## 数据存储位置

### 1. 数据库（MySQL）

**表名：** `test_case_suites`

**关键字段：**
- `id`: 主键
- `project_id`: 项目ID
- `name`: 场景用例集名称（如："用户管理"、"订单管理"）
- `description`: 场景描述
- `test_case_ids`: JSON格式的接口ID列表，格式：`["__LOGIN_INTERFACE__", "28", "29", ...]`
- `tags`: 标签（逗号分隔）
- `created_at`: 创建时间
- `updated_at`: 更新时间

### 2. Redis

**Key格式：** `project:{project_id}:scenarios`

**数据结构：**
```json
{
  "scenarios": [
    {
      "scenario_name": "用户管理",
      "dependency_chain": ["28", "29", "30"],
      "analysis_summary": "场景描述..."
    }
  ],
  "total_count": 10,
  "interfaces_count": 50,
  "login_interface": {
    "name": "用户登录",
    "method": "POST",
    "path": "/V0.1/index.php",
    ...
  }
}
```

**过期时间：** 30天

### 3. Neo4j（可选）

存储场景用例集的图结构数据，用于知识图谱展示和查询。

### 4. ChromaDB（可选）

存储场景用例集的向量数据，用于相似度检索和RAG增强。

## 场景用例集的内容

每个场景用例集包含：

1. **登录接口**（特殊标识 `__LOGIN_INTERFACE__`）
   - 每个场景的第一个接口都是登录接口
   - 用于获取token，供后续接口使用

2. **业务接口列表**（按依赖顺序排列）
   - 按照CRUD操作顺序：创建(Create) -> 修改(Update) -> 查询(Read) -> 删除(Delete)
   - 基于接口依赖分析结果排序
   - 过滤掉账号、oauth、小度、阿里云等相关接口

3. **场景名称和描述**
   - 名称通常基于接口分组（如："用户管理"、"订单管理"）
   - 描述包含场景的业务逻辑说明

## 与测试用例生成的关系

场景用例集生成后，可以：

1. **生成场景测试用例**
   - 在"小场景用例集"页面点击"生成接口场景用例"按钮
   - 调用 `POST /api/test-case-suites/{suite_id}/generate-test-cases?case_type=pytest&generate_type=scenario`
   - 生成一个场景测试用例（`TestCase`），名称格式：`{suite.name}_场景测试用例`

2. **生成接口测试用例**
   - 在"小场景用例集"页面点击"生成接口用例"按钮（如果支持）
   - 调用 `POST /api/test-case-suites/{suite_id}/generate-test-cases?case_type=pytest&generate_type=interface`
   - 为场景用例集中的每个接口生成独立的接口测试用例

3. **创建测试任务**
   - 在"测试任务"页面创建测试任务时，可以选择场景用例集
   - 系统会根据场景用例集中的接口列表，找到对应的测试用例进行执行

## 总结

**小场景测试用例集的数据生成时机：**

1. **主要方式（自动生成）：** 运行"全局接口依赖分析"时，在分析过程中自动生成并存储到数据库
2. **辅助方式（手动生成）：** 在"小场景用例集"页面手动点击"从依赖分析结果生成小场景用例集"按钮

**建议流程：**
1. 导入接口文档
2. 运行"全局接口依赖分析"（会自动生成场景用例集）
3. 在"小场景用例集"页面查看生成的场景用例集
4. 选择场景用例集，生成测试用例
5. 创建测试任务并执行
