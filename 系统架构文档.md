## 系统概述

API接口智能测试系统是一个基于大模型的智能API测试平台，采用前后端分离的微服务架构，通过Docker Compose进行容器化部署。

### 核心特性

- **智能文档解析**：支持OpenAPI/Swagger、JSON、YAML、PDF、Word等多种格式
- **AI驱动的测试用例生成**：基于DeepSeek和通义千问大模型自动生成测试用例
- **接口依赖分析**：自动分析接口间的依赖关系和业务逻辑
- **数据库连接分析**：连接业务数据库，分析数据结构，辅助测试数据生成
- **异步任务处理**：使用Celery处理耗时任务（文档解析、用例生成、测试执行）
- **知识图谱**：使用Neo4j存储和可视化接口关系网络
- **向量检索**：使用Milvus进行语义搜索和相似度匹配
- **测试执行与报告**：支持pytest和JMeter两种测试框架，生成Allure报告

## 技术架构

### 架构模式

- **前后端分离架构**：React前端 + FastAPI后端
- **微服务架构**：基于Docker Compose的多容器服务
- **异步任务处理**：Celery + Redis任务队列
- **RESTful API**：标准化的API接口设计
- **分层架构**：路由层 → 服务层 → 数据访问层

### 技术选型

| 层级 | 技术栈 |
|------|--------|
| **前端** | React 18 + Ant Design 5 + ECharts + Monaco Editor |
| **后端** | FastAPI + SQLAlchemy 2.0 + Pydantic 2.0 |
| **数据库** | MySQL 8.0（主数据库） |
| **缓存** | Redis 7（任务队列 + 缓存） |
| **向量数据库** | Milvus 2.3（语义搜索） |
| **图数据库** | Neo4j 5.15（知识图谱） |
| **对象存储** | MinIO（文件存储） |
| **任务队列** | Celery 5.3 + Redis |
| **大模型** | DeepSeek API + 通义千问API |
| **测试框架** | pytest + JMeter |
| **报告生成** | Allure + 自定义HTML报告 |

## 数据架构

### 数据库设计

#### 核心数据表

1. **users** - 用户表
   - 存储用户基本信息（用户名、邮箱、密码哈希）

2. **projects** - 项目表
   - 存储项目信息（名称、描述、所属用户）

3. **documents** - 文档表
   - 存储上传的API文档信息（文件名、类型、路径、解析结果）

4. **api_interfaces** - API接口表
   - 存储手动添加的API接口信息

5. **document_api_interfaces** - 文档解析的API接口表
   - 存储从文档中解析出的详细接口信息
   - 包含请求/响应头、请求体、响应体、Schema等

6. **test_cases** - 测试用例表
   - 存储生成的测试用例（代码、数据、断言、依赖关系）

7. **test_environments** - 测试环境表
   - 存储测试环境配置（base_url、登录信息、xjid等）

8. **test_case_suites** - 测试用例集合表
   - 存储测试用例分组信息

9. **test_tasks** - 测试任务表
   - 存储测试执行任务（状态、进度、结果摘要）

10. **test_results** - 测试结果表
    - 存储每个测试用例的执行结果

11. **test_debug_records** - 测试调试记录表
    - 存储测试用例的调试执行记录

12. **db_connections** - 数据库连接表
    - 存储业务数据库连接信息（用于数据分析和测试数据生成）

### 数据关系图

```
users
  └── projects (1:N)
       ├── documents (1:N)
       │    └── document_api_interfaces (1:N)
       ├── api_interfaces (1:N)
       ├── test_cases (1:N)
       │    └── api_interfaces (N:1)
       ├── test_environments (1:N)
       ├── test_case_suites (1:N)
       ├── test_tasks (1:N)
       │    ├── test_environments (N:1)
       │    ├── test_case_suites (N:1)
       │    └── test_results (1:N)
       │         └── test_cases (N:1)
       └── db_connections (1:N)
```

### 向量数据库（Milvus）

- **集合（Collection）**：`api_interfaces`
- **向量维度**：根据使用的embedding模型确定
- **用途**：
  - 存储API接口的向量表示
  - 支持语义搜索和相似度匹配
  - 用于RAG检索

### 图数据库（Neo4j）

- **节点类型**：
  - `Interface` - API接口节点
  - `Database` - 数据库节点
  - `Table` - 数据表节点
  - `Field` - 字段节点

- **关系类型**：
  - `DEPENDS_ON` - 接口依赖关系
  - `CALLS` - 接口调用关系
  - `USES` - 接口使用数据库表
  - `CONTAINS` - 数据库包含表，表包含字段

### Docker Compose架构

系统使用Docker Compose进行容器化部署，所有服务运行在同一个Docker网络中。

### 网络配置

- **网络名称**：`api_test_network`
- **网络类型**：bridge
- **服务间通信**：通过服务名称（如`mysql`、`redis`等）

- `mysql_data` - MySQL数据持久化
- `redis_data` - Redis数据持久化（如果需要）

## 数据流

### 文档解析流程

```
用户上传文档
    ↓
Frontend → Backend API
    ↓
Backend创建Document记录
    ↓
Backend发送Celery任务（parse_document_task）
    ↓
Celery Worker接收任务
    ↓
DocumentParser解析文档
    ↓
使用LLM进行智能理解
    ↓
提取API接口信息
    ↓
存储到document_api_interfaces表
    ↓
更新Document状态为"completed"
```

### 测试用例生成流程

```
用户选择接口生成测试用例
    ↓
Frontend → Backend API
    ↓
Backend创建TestCase记录（status="generating"）
    ↓
Backend发送Celery任务（generate_test_case_task）
    ↓
Celery Worker接收任务
    ↓
TestCaseGenerator生成测试代码
    ↓
使用LLM增强测试用例
    ↓
更新TestCase记录（test_code、status="completed"）
    ↓
前端轮询获取生成结果
```

### 测试执行流程

```
用户创建测试任务
    ↓
Frontend → Backend API
    ↓
Backend创建TestTask记录
    ↓
分析接口依赖关系
    ↓
排序测试用例（按依赖关系）
    ↓
Backend发送Celery任务（execute_test_task）
    ↓
Celery Worker接收任务
    ↓
TestExecutor执行测试
    ├── pytest执行（接口测试）
    └── JMeter执行（性能测试）
    ↓
收集测试结果
    ↓
生成测试报告（Allure/HTML）
    ↓
使用LLM分析性能测试结果（如果是性能测试）
    ↓
更新TestTask状态和结果
```

### 知识图谱构建流程

```
接口依赖分析
    ↓
RelationshipAnalyzer分析关系
    ↓
存储到Neo4j
    ├── 创建Interface节点
    ├── 创建DEPENDS_ON关系
    └── 创建CALLS关系
    ↓
向量化接口信息
    ↓
存储到Milvus
    ↓
前端查询知识图谱
    ↓
Neo4j查询关系数据
    ↓
返回给前端可视化
```

## 安全架构

### 认证与授权

- **JWT Token认证**：
  - 使用HS256算法
  - Token过期时间：1440分钟（24小时）
  - 存储在HTTP请求头：`Authorization: Bearer <token>`

- **密码安全**：
  - 使用bcrypt进行密码哈希
  - 密码不存储明文

### API安全

- **CORS配置**：
  - 开发环境：允许所有来源
  - 生产环境：明确指定允许的来源

- **输入验证**：
  - 使用Pydantic进行数据验证
  - SQL注入防护（使用SQLAlchemy ORM）

### 数据安全

- **数据库连接**：
  - 使用环境变量存储敏感信息
  - 不在代码中硬编码密码

- **文件上传**：
  - 限制文件大小（100MB）
  - 验证文件类型
  - 文件存储在隔离的目录

### 网络安全

- **Docker网络隔离**：
  - 所有服务运行在独立的Docker网络中
  - 服务间通过服务名称通信
  - 仅暴露必要的端口

- **端口映射**：
  - 使用非标准端口避免冲突
  - 生产环境建议使用反向代理（Nginx）
