# API接口智能测试系统 - 面试话术

## 一句话架构

前端只需上传一份接口文档（OpenAPI/Swagger/PDF/Word），后端服务的多智能体进行分工：**文档解析智能体**使用多模态大模型（通义千问/DeepSeek）先解析文档，提取所有接口信息；接着**依赖分析智能体**分析接口间的调用关系、数据流依赖和业务逻辑依赖；然后**测试用例生成智能体**基于接口信息和依赖关系，使用Few-shot学习和RAG增强，生成pytest或JMeter格式的测试用例代码；最后通过Celery异步任务执行测试并生成Allure报告，整个过程全自动化，用户只需上传文档即可得到可执行的测试用例。

---

## 三分钟实现逻辑描述

### 【1.5分钟】核心业务流程

**第一步：文档上传与解析（10秒）**

用户在前端上传接口文档（OpenAPI/Swagger JSON/YAML、PDF、Word等），前端将文件上传到后端，后端创建Document记录并发送Celery异步任务（`parse_document_task`）。

**第二步：多智能体协作分析（70秒）**

首先是**文档解析智能体（DocumentParser）**，它根据文档类型选择不同的解析策略：
- 对于OpenAPI/Swagger格式：直接JSON/YAML解析，提取接口路径、方法、参数、响应等结构化信息
- 对于PDF/Word文档：使用多模态大模型（通义千问VL）进行视觉解析，将文档转换为图片，AI识别图片中的接口信息并提取为结构化数据
- 所有解析结果都会使用LLM进行标准化处理，确保数据格式统一

解析完成后，接口信息存储到`document_api_interfaces`表。

接着是**接口依赖分析智能体（DependencyAnalyzerAgent）**，它分析接口之间的依赖关系：
- **调用依赖**：识别哪些接口需要先调用其他接口（如登录接口返回token，其他接口需要token）
- **数据流依赖**：分析参数传递关系（如创建订单返回order_id，查询订单需要order_id）
- **业务逻辑依赖**：使用LLM分析业务逻辑，确定接口执行顺序（如必须先注册才能登录，必须先创建才能删除）
- **版本号规则**：严格区分不同版本的接口，避免混淆

依赖分析采用多层策略：
1. **参数匹配**：分析请求参数和响应字段的匹配关系
2. **URL模式匹配**：识别路径参数依赖
3. **LLM智能分析**：使用DeepSeek模型分析复杂的业务逻辑依赖
4. **数据库关系分析**：结合Neo4j知识图谱中的数据库表关系分析依赖

分析结果会存储到Neo4j图数据库，构建接口关系网络。

然后是**测试用例生成智能体（TestCaseGeneratorAgent）**，它基于接口信息和依赖关系生成测试用例：
- **Few-shot学习**：从Redis缓存或数据库中获取历史高质量测试用例作为示例
- **RAG增强生成**：
  - 从Milvus向量库中检索相似接口的测试用例
  - 从Neo4j知识图谱中查询接口依赖关系
  - 将这些上下文信息一起传递给LLM，提升生成质量
- **场景化生成**：基于依赖关系生成完整的业务场景测试用例（如"用户下单流程"：登录→查看商品→创建订单→查询订单）
- **多格式支持**：可以生成pytest格式的功能测试用例，也可以生成JMeter格式的性能测试脚本

生成的测试用例代码存储到`test_cases`表。

**第三步：测试执行与报告生成（10秒）**

用户选择测试用例创建测试任务，系统通过Celery异步执行：
- **依赖排序**：根据依赖分析结果，使用拓扑排序确定测试用例执行顺序
- **数据准备**：自动生成测试数据，处理接口间的数据传递（如token、ID等）
- **测试执行**：使用pytest执行功能测试，或使用JMeter执行性能测试
- **结果收集**：收集测试结果，生成Allure测试报告
- **智能分析**：对于性能测试，使用LLM分析测试结果，识别性能瓶颈并提供优化建议

---

### 【1分钟】关键技术实现

**1. 多智能体协作机制**

使用LangGraph框架构建多智能体工作流。每个智能体对应一个节点：
- `InterfaceParserAgent`：解析节点，负责文档解析和接口提取
- `DependencyAnalyzerAgent`：依赖分析节点，负责分析接口依赖关系
- `TestCaseGeneratorAgent`：用例生成节点，负责生成测试用例

工作流定义：
```
parser → dependency_analyzer → testcase_generator → END
```

每个智能体处理完成后，将结果传递到下一个节点，最终输出完整的测试用例。

**2. 异步任务处理机制**

使用Celery + Redis实现异步任务处理：
- **任务类型**：
  - `parse_document_task`：文档解析（30分钟超时）
  - `generate_test_case_task`：测试用例生成（30分钟超时）
  - `generate_scenario_test_case_task`：场景测试用例生成（40分钟超时）
  - `execute_test_task`：测试执行（60分钟超时）
- **状态管理**：支持PENDING、PROGRESS、SUCCESS、FAILURE状态
- **进度更新**：通过`update_state`实时更新任务进度，前端轮询获取
- **断点续传**：长时间任务支持checkpoint机制，任务中断后可恢复

**3. 提示词工程**

针对每个智能体设计了专业的提示词模板：

- **文档解析提示词**：强调提取完整的接口信息（名称、方法、URL、参数、响应格式等），输出标准化JSON格式
- **依赖分析提示词**：引导LLM分析调用依赖、数据流依赖、业务逻辑依赖，结合数据库关系进行综合分析
- **用例生成提示词**：融入pytest/JMeter最佳实践，强调生成高质量、可维护的测试代码，包含正常、异常、边界场景

**4. RAG（检索增强生成）机制**

- **向量检索（Milvus）**：将接口信息向量化存储，生成用例时检索相似接口的测试用例作为参考
- **知识图谱检索（Neo4j）**：查询接口依赖关系，确保生成的用例考虑依赖关系
- **混合检索（HybridRAGService）**：结合向量检索和图检索，提供更丰富的上下文信息

**5. AI模型智能选择**

根据任务类型选择最适合的模型：
- **文档解析（PDF/Word）**：使用通义千问VL多模态模型进行视觉理解
- **文本生成（用例生成）**：使用DeepSeek模型生成高质量代码
- **依赖分析**：使用DeepSeek模型分析业务逻辑
- **性能分析**：使用DeepSeek模型分析性能测试结果

### 【30秒】总结

整个平台的核心创新在于**多智能体协作**和**智能依赖分析**。通过将复杂任务分解为文档解析、依赖分析、用例生成三个专业智能体，每个智能体专注单一职责，配合专业的提示词引导和RAG增强，AI能够准确理解接口文档并生成高质量的测试用例。基于依赖关系分析的场景化用例生成，确保测试用例按正确顺序执行，覆盖完整的业务流程。整个流程完全自动化，用户只需上传文档，系统就能自动生成可执行的测试用例并执行测试。

## 详细智能体工作说明

### 总体一句话：

用户给一份接口文档（OpenAPI/PDF/Word），**文档解析智能体**先把文档里所有接口的信息都提取出来（方法、URL、参数、响应等）；**依赖分析智能体**分析接口之间的调用关系、数据传递关系和业务逻辑依赖；**测试用例生成智能体**基于接口信息和依赖关系，使用Few-shot学习和RAG增强，生成pytest或JMeter格式的可执行测试用例代码；最后通过Celery异步执行测试并生成报告，用例就出来了。

### 1. 文档解析智能体：帮你"读懂接口文档"

你可以把它想象成一个特别会看文档的测试同事：

- 你给它一份**接口文档**（OpenAPI JSON、Swagger YAML、PDF、Word等）。
- 它根据文档类型选择不同的解析策略，大致在干三件事：

  **1. 识别文档类型并选择解析器**
  - **OpenAPI/Swagger（JSON/YAML）**：直接解析JSON/YAML结构，提取paths、methods、parameters、responses等
  - **PDF/Word文档**：使用多模态大模型（通义千问VL）进行视觉解析
    - 将PDF转换为图片（每页一张）
    - 将Word文档转换为图片或直接提取文本
    - AI识别图片中的接口信息（接口名称、HTTP方法、URL、参数、响应格式等）
  - **Postman Collection**：解析JSON格式的Postman导出文件
  - **Excel表格**：解析表格格式的接口清单

  **2. 提取接口信息并标准化**
  每个接口提取以下信息：
  - **接口名称**：从summary或title中提取
  - **HTTP方法**：GET、POST、PUT、DELETE等
  - **URL路径**：接口的路径（如`/api/users/{id}`）
  - **请求参数**：
    - 查询参数（query parameters）
    - 路径参数（path parameters）
    - 请求体（request body）
  - **请求头**：Content-Type、Authorization等
  - **响应格式**：状态码、响应体结构、响应头
  - **接口描述**：接口的功能说明

  **3. 使用LLM进行智能理解和标准化**
  - 将提取的原始数据传递给LLM（DeepSeek）
  - LLM根据标准化的Schema进行理解和转换
  - 输出统一格式的JSON数据，存储到`document_api_interfaces`表

可以理解为：**它只做一件事——把各种格式的接口文档"翻译"成系统能识别的结构化接口数据**。

**实际例子**：

输入：PDF文档（包含接口描述）
```
用户登录接口
POST /api/login
请求参数：username, password
响应：{code: 200, data: {token: "xxx"}}
```

输出：结构化JSON
```json
{
  "interfaces": [
    {
      "name": "用户登录",
      "method": "POST",
      "url": "/api/login",
      "body": {
        "username": "string",
        "password": "string"
      },
      "response_schema": {
        "code": 200,
        "data": {
          "token": "string"
        }
      },
      "description": "用户登录接口"
    }
  ]
}
```

### 2. 依赖分析智能体：帮你"理清接口关系"

有了"接口清单"之后，第二个智能体更像是**架构分析师**：

- 它的输入是：
  1. 文档解析智能体输出的**接口列表**（所有接口的信息）
  2. 项目的**数据库连接信息**（可选，用于分析数据库关系）
  3. 已有的**知识图谱数据**（Neo4j中存储的历史依赖关系）

- 它要做的是：分析接口之间的依赖关系，比如：
  1. **调用依赖**：接口B需要先调用接口A（如获取用户信息需要先登录获取token）
  2. **数据流依赖**：接口A的响应数据被接口B使用（如创建订单返回order_id，查询订单需要order_id）
  3. **业务逻辑依赖**：接口的执行顺序有业务要求（如必须先注册才能登录，必须先创建才能删除）
  4. **版本号规则**：严格区分不同版本的接口（如`/api/v1/users`和`/api/v2/users`）

- 具体分析策略：

  **策略1：参数匹配分析**
  - 分析接口A的响应字段和接口B的请求参数
  - 如果字段名相同或相似（如`user_id`、`order_id`），识别为数据依赖
  - 如果接口B的路径参数在接口A的响应中出现，识别为路径参数依赖

  **策略2：LLM智能分析**
  - 对于复杂的业务逻辑依赖，使用DeepSeek模型分析
  - 提示词引导AI分析接口的业务含义和执行顺序
  - AI能识别隐式的业务逻辑依赖（如"查询订单"必须在"创建订单"之后）

  **策略3：数据库关系分析**
  - 结合Neo4j知识图谱中的数据库表关系
  - 如果接口操作的表存在外键关系，识别为数据依赖
  - 例如：订单表依赖用户表，所以"创建订单"接口依赖"创建用户"接口

  **策略4：版本号规则**
  - 严格区分不同版本的接口
  - 不同版本的接口不会产生依赖关系
  - 例如：`/api/v1/users`和`/api/v2/users`是独立的

- 输出结果：
  - **依赖关系图**：节点（接口）+ 边（依赖关系）
  - **拓扑排序结果**：确定接口执行顺序
  - **依赖链**：完整的调用链（如：登录→获取用户信息→创建订单→查询订单）

用大白话说：**这个智能体负责把"有哪些接口"，变成"接口之间有什么关系、应该按什么顺序执行"的关系图**。

**实际例子**：

输入：接口列表
```json
[
  {"id": 1, "name": "用户登录", "method": "POST", "url": "/api/login", "response": {"token": "string"}},
  {"id": 2, "name": "获取用户信息", "method": "GET", "url": "/api/user/{user_id}", "headers": {"Authorization": "Bearer {token}"}},
  {"id": 3, "name": "创建订单", "method": "POST", "url": "/api/orders", "headers": {"Authorization": "Bearer {token}"}}
]
```

分析过程：
1. 接口2需要`Authorization` header，可能来自接口1的响应`token` → **认证依赖**
2. 接口2需要`user_id`路径参数，可能来自接口1的响应或需要先创建用户 → **数据依赖**
3. 接口3需要`Authorization` header，来自接口1 → **认证依赖**
4. 业务逻辑上，获取用户信息和创建订单都需要先登录 → **业务逻辑依赖**

输出：依赖关系
```json
{
  "nodes": [
    {"id": 1, "name": "用户登录"},
    {"id": 2, "name": "获取用户信息"},
    {"id": 3, "name": "创建订单"}
  ],
  "edges": [
    {
      "source": 1,
      "target": 2,
      "type": "auth_dependency",
      "data_flow": "token: response.data.token -> header.Authorization"
    },
    {
      "source": 1,
      "target": 3,
      "type": "auth_dependency",
      "data_flow": "token: response.data.token -> header.Authorization"
    }
  ],
  "topological_order": [1, 2, 3]  // 执行顺序：先登录，再获取用户信息，最后创建订单
}
```

### 3. 测试用例生成智能体：把"接口信息+依赖关系"翻译成可执行的测试代码

前两个智能体输出的，都是**"人能看懂"的结构化描述**，还不能直接跑。  
测试用例生成智能体负责把这些信息转换成**真正能执行的测试代码**。

- 输入：
  1. **接口信息**（文档解析智能体的输出）
  2. **依赖关系**（依赖分析智能体的输出）
  3. **Few-shot示例**（从Redis或数据库中获取的历史高质量测试用例）
  4. **RAG上下文**（从Milvus向量库检索的相似接口用例，从Neo4j查询的依赖关系）

- 生成策略：

  **策略1：Few-shot学习**
  - 从Redis缓存或数据库中获取3-5个高质量测试用例作为示例
  - 这些示例展示了标准的测试用例格式、断言方式、数据构造方法
  - LLM参考这些示例，生成风格一致、质量高的测试用例

  **策略2：RAG增强生成**
  - **向量检索（Milvus）**：检索相似接口的测试用例作为参考
    - 例如：要生成"用户登录"接口的用例，检索"管理员登录"、"手机号登录"等相似接口的用例
    - 这些用例提供了可参考的测试场景、断言方式、数据构造方法
  - **知识图谱检索（Neo4j）**：查询接口依赖关系
    - 例如：查询"获取用户信息"接口依赖"用户登录"接口
    - 在生成用例时，考虑依赖关系，生成包含前置步骤的完整场景用例

  **策略3：场景化生成**
  - 基于依赖关系生成完整的业务场景测试用例
  - 例如："用户下单流程"用例：
    1. 先调用登录接口获取token
    2. 调用查看商品列表接口获取商品ID
    3. 调用创建订单接口（使用token和商品ID）
    4. 调用查询订单接口验证订单创建成功
  - 这种场景化用例测试完整的业务流程，而不只是单个接口

  **策略4：多场景覆盖**
  - **正常场景**：使用有效数据测试接口正常功能
  - **异常场景**：测试错误处理（如密码错误、参数缺失等）
  - **边界值场景**：测试边界条件（如字符串长度限制、数值范围等）
  - **依赖场景**：测试接口依赖关系的正确处理

- 输出格式：

  **pytest格式**：
  ```python
  import pytest
  import requests
  import allure

  @allure.feature("用户登录")
  class TestLogin:
      """用户登录接口测试"""

              "username": "testuser",

          assert response.json()["code"] == 200
          assert "token" in response.json()["data"]
      
      @allure.story("密码错误")
      def test_login_wrong_password(self):
          """测试密码错误场景"""
          # ...测试代码
  ```

  **JMeter格式（性能测试）**：

用大白话说：**这个智能体负责把"接口信息+依赖关系"翻译成"真正能跑的测试代码"，还参考了历史用例和相似接口，确保生成的代码质量高、风格统一**。

### 4. 三个智能体串起来，就是"自动写用例"的完整链路

用一句更顺的"大白话"串一下整体过程：

1. **文档解析智能体**：
   "先让我看看这份文档里**有什么接口**，OpenAPI我直接解析，PDF/Word我用AI看图片识别，最后给你整理成一份**接口清单**，每个接口的方法、URL、参数、响应格式我都给你写清楚。"

2. **依赖分析智能体**：
   "你告诉我接口清单后，我帮你分析**接口之间的关系**，哪些接口需要先调用其他接口（比如需要token）、哪些接口的数据会被其他接口使用（比如order_id）、接口应该按什么顺序执行，我都给你理清楚，还存到Neo4j知识图谱里。"

3. **测试用例生成智能体**：
   "你这套接口信息和依赖关系我看懂了，我再从历史用例库里找几个**高质量示例**，从向量库找**相似接口的用例**，从知识图谱查**依赖关系**，然后帮你翻译成**pytest的Python代码**或者**JMeter的性能测试脚本**，保证结构、格式都符合规范，可以直接丢给测试框架跑。"

所以，用例是怎么生成的？
- 不是直接从文档一下子变代码，
- 而是按照"**文档 → 接口清单 → 依赖关系图 → 测试用例代码**"这样一步步细化出来的，
- 每一步都有专门的智能体负责一个小环节，最后由测试用例生成智能体把前面所有信息"汇总翻译"成真正可执行的测试代码。

## 面试口述版1分钟回答稿

**问：请介绍一下你的API接口智能测试系统项目？**

**答**：

这是一个基于大模型的智能API测试平台，核心是用AI自动生成测试用例。

**一句话概括**：用户上传接口文档，系统通过三个智能体协作，自动解析文档、分析接口依赖关系、生成可执行的测试用例代码。

**具体流程**：

1. **文档解析**：用户上传OpenAPI文档或PDF/Word格式的接口文档，文档解析智能体使用多模态大模型解析文档，提取所有接口的结构化信息（方法、URL、参数、响应格式等）。

2. **依赖分析**：依赖分析智能体分析接口之间的关系，比如哪些接口需要先调用其他接口获取token，哪些接口的数据会被其他接口使用，接口应该按什么顺序执行。分析结果存储到Neo4j知识图谱。

3. **用例生成**：测试用例生成智能体基于接口信息和依赖关系，使用Few-shot学习（参考历史高质量用例）和RAG增强（检索相似接口用例），生成pytest格式的功能测试用例或JMeter格式的性能测试脚本。

4. **测试执行**：通过Celery异步任务执行测试，自动处理依赖关系和数据传递，生成Allure测试报告。对于性能测试，还会使用LLM分析测试结果，识别性能瓶颈。

**技术亮点**：
- 多智能体协作（使用LangGraph编排流程）
- 智能依赖分析（结合规则引擎和LLM）
- RAG增强生成（向量检索+知识图谱）
- Few-shot学习（参考历史用例提升质量）
- 场景化生成（基于依赖关系生成完整业务流程用例）

整个过程完全自动化，用户只需上传文档，系统就能自动生成可执行的测试用例并执行测试，大大提升了测试效率。
